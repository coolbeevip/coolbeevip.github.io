# AI-first 柔性制造探索，第 6 章：换型准备链路

摘要：这里讨论的不是在线动作控制，而是产线本来就有的换型准备流程。`Agentic AI` 不替代 `MES`、`PLC` 或站控放行。它主要用来把新旧配置差异、风险点和补充检查项整理清楚，减少现场对个人经验的依赖。

## 场景定义

在 3C 小型精密装配的固定工位里，换型通常由新工单、新料号、新配方或新夹具触发。真正决定能否放行的，仍然是传统工业链路里的换型准备逻辑：

* `MES` 或配方系统下发新工单
* 工位比较当前状态和目标状态是否一致
* 若存在料号、配方、夹具、相机参数或工位 ready 状态不一致，则进入换型准备
* 满足放行条件后，再由主协调单元放行

问题不在于“AI 会不会自动猜到要换型”，而在于传统换型流程能不能借它变得更清楚、更稳。

## 链路拆解

### 1. 触发换型准备

换型通常靠结构化比较触发，不靠自然语言理解。最常见的比较项包括：

* 新工单料号与当前工位料号是否一致
* 配方版本是否一致
* 夹具和末端工具是否已切换
* 相机参数是否与当前产品版本匹配
* 当前工位是否处于可进入换型准备的 ready 状态

这些条件里只要有一项不一致，换型准备就该启动。

### 2. 传统流程完成基础核对

在没有 `Agentic AI` 的情况下，站控或换型逻辑通常会完成这些动作：

* 工单解析
* 配方匹配
* 参数核对
* 风险分层
* 放行资格检查

这一层本质上还是规则驱动的工业逻辑，属于主链路。

这套流程虽然能跑，但在多配方、小批量、高频换型的现场里，几个老问题会很快冒出来：

* 检查项分散在 `MES`、`HMI`、`SOP` 和现场经验里，新人很难一次看全
* 风险分层往往比较粗，哪些差异真正影响首件良率，很多时候还是靠工程师自己判断
* 补充检查项不稳定，同样一次换型，不同班次和不同工程师关注点可能不一样
* 结果难沉淀，这次换型踩过的坑，下次未必能被系统主动提醒
* 低风险和高风险的边界经常没有提前定清，最后容易演变成“全靠人工多看一遍”

平时这些问题可能只是零散出现。换型频率一高，或者现场经验不够，它们就会直接反映在准备时间、首件波动和人工负担上。

### 3. Agentic AI 做差异增强

`Agentic AI` 在这里读的是换型上下文，不拿底层控制权。典型输入包括：

* 新旧机种或新旧料号差异
* 当前夹具、吸嘴、末端工具状态
* 当前视觉参数和相机配置
* 历史异常与首件风险记录

基于这些输入，它更适合输出：

* 风险等级
* 高风险项排序
* 补充检查项
* 首件验证关注点
* 需要人工确认的项目清单

### 4. 人工只在高风险点介入

人工确认不应成为每次换型的默认路径。通常会这样分：

* 低风险换型自动放行
* 高风险换型要求人工确认
* 首件验证或规则明确要求时再进入 `HMI` 确认页

这样不会把“换型助手”又做成一个新的人工卡点。

## 这个场景里 Agentic AI 的真实位置

这个场景最容易被误解的地方，是把 `Agentic AI` 当成直接参与换型执行的控制层。更合理的位置是：

* 上接 `MES`、配方系统、`SOP` 和历史案例
* 横向接 `HMI`、回放系统和知识维护入口
* 下接传统换型准备逻辑的结构化输入输出
* 最终是否放行，仍由主协调单元裁决

它更像换型助手，不是换型控制器。

## 首版系统最值得做的增强点

如果这一块只做一版最小产品，最好别停在“应该做差异分析”这种抽象表述上。更有用的写法，是直接把增强点落到一组匿名化示例数据上。下面这种表达方式，比较适合写进方案或复盘材料：

> 以下数据为匿名化示例，用来说明首版系统该优先解决什么问题。

以一个 3C 固定工位的月度换型记录为例，假设一个月内发生了 `48` 次换型，其中：

* `19` 次出现配方版本不一致
* `11` 次出现夹具型号或吸嘴配置不一致
* `7` 次出现视觉参数未切换到目标产品版本
* `14` 次需要工程师额外补充检查项
* `9` 次首件复核后发现前序检查有遗漏

顺着这组数据往下看，首版系统优先要补的通常不是“大模型自动决策”，而是几件更具体的事情：

| 观察到的问题 | 首版优先增强点 |
| --- | --- |
| 配方、夹具、视觉参数不一致最常见 | 把换型差异做成结构化比较，统一输出差异清单 |
| 工程师经常临时补充检查项 | 把补充检查项沉淀成可维护模板，并按产品版本自动带出 |
| 首件复核还能发现遗漏 | 把高风险项排序，并明确首件验证关注点 |
| 一部分低风险换型也被拉进人工复核 | 提前定清低风险自动放行和高风险人工确认的边界 |
| 现场只能说“这次又出过问题” | 把首件验证结论和最终结果关联起来，形成可追溯记录 |

这里想说的其实很简单：首版系统先解决高频、重复、容易遗漏的准备问题，比一开始追求复杂推理更值。先把这些基础数据和检查逻辑接稳，换型准备才可能从经验流程变成能持续积累的产品能力。

## 这一节如果真的要做，通常怎么落地

前面讲的是为什么值得做。真到实现时，先把一条边界守住：不要让大语言模型直接决定工位放不放行。

现场更常见的做法是四层：

* 规则层先判断是否进入换型准备
* 上下文组装层把配方差异、工位状态和历史案例整理成结构化输入
* 大语言模型只生成风险判断、补充检查项和首件关注点
* 最终是否放行，仍由规则层、站控或主协调单元裁决

大语言模型给的是建议包，不是最终裁决。

### 1. 什么时候值得调用大语言模型

不是每次换型都要调模型。通常先用规则筛一遍：

* 如果只是低风险差异，而且规则覆盖完整，就直接放行
* 如果差异组合复杂、历史上出过问题，或者需要动态补充检查项，再调用大语言模型
* 如果当前状态本身不满足基本资格条件，例如工位不 ready、夹具未装到位、关键参数未加载，就先拦截，不进入模型推理

原因不复杂。换型准备里有不少问题本来就适合用结构化规则处理，没必要把所有事情都交给大语言模型。

### 2. 配方系统的数据最好长什么样

配方系统不是字段越多越好，关键是字段要能比较。至少应该有下面这些信息：

```json
{
  "recipe_id": "摄像头环_v2",
  "product_family": "手机摄像头环",
  "fixture_model": "夹具_12",
  "nozzle_model": "吸嘴_B1",
  "vision_profile": "视觉参数_21",
  "motion_profile": "低速动作_v3",
  "inspection_profile": "摄像头环标准检测",
  "qualification_flags": {
    "first_article_required": true,
    "manual_confirm_required": false
  }
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `recipe_id` | 目标配方编号，用来唯一标识当前要切换到哪套工艺参数。 |
| `product_family` | 产品族或机种类别，用来决定可复用哪些检查模板和历史案例。 |
| `fixture_model` | 当前产品应使用的夹具型号。 |
| `nozzle_model` | 当前产品应使用的吸嘴或末端工具型号。 |
| `vision_profile` | 视觉参数配方编号，例如曝光、模板、阈值等配置集合。 |
| `motion_profile` | 动作或速度相关配方编号，用来区分不同产品使用的运动参数。 |
| `inspection_profile` | 首件或过程检验所使用的检测配置编号。 |
| `qualification_flags.first_article_required` | 是否要求首件验证。 |
| `qualification_flags.manual_confirm_required` | 是否要求人工确认后才能放行。 |

这些字段不是为了把“原始配置”直接丢给模型，而是为了让系统先做结构化差异比对。

### 3. 历史案例数据最好怎么存

历史案例如果只是一堆日志文本，后面很难真正用起来。至少要收成下面这种结构：

```json
{
  "case_id": "chg_1182",
  "station_type": "摄像头环装配",
  "product_family": "手机摄像头环",
  "diff_pattern": ["视觉参数", "吸嘴型号"],
  "issue": "首件偏移过高",
  "resolution": "人工确认 + 首件复核",
  "result": "调整后通过",
  "first_article_pass": true,
  "manual_confirm_required": true
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `case_id` | 历史换型案例编号，用来追溯具体记录。 |
| `station_type` | 工位类型，用来过滤掉不相关工位的案例。 |
| `product_family` | 案例所属产品族，用来找同类产品的历史换型经验。 |
| `diff_pattern` | 当次换型涉及的主要差异项组合，例如视觉参数变化、吸嘴变化。 |
| `issue` | 当次换型或首件阶段出现的主要问题。 |
| `resolution` | 现场最后采用的处理方式。 |
| `result` | 处理后的最终结果，例如调整后通过、转人工、放弃自动放行。 |
| `first_article_pass` | 首件是否通过。 |
| `manual_confirm_required` | 该案例是否最终需要人工确认。 |

这样后面才能做相似案例检索，而不是把大段历史日志全文塞进提示词里。

### 4. 一个更实用的上下文通常怎么组成

大语言模型看到的上下文，最好是系统整理过的一包结构化对象，而不是 `MES`、`SOP`、日志全文和截图的混合体。一个能实际落地的上下文，通常由下面几部分组成：

* 当前工位信息
* 当前工单和目标配方
* 当前已加载配方和现场实际状态
* 系统预先计算出的差异项
* 相似历史案例
* 企业或站级规则

可以长成这样：

```json
{
  "station": {
    "station_id": "3c_cell_01",
    "station_type": "摄像头环装配",
    "current_state": "换型准备就绪"
  },
  "work_order": {
    "order_id": "wo_20260407_01",
    "product_family": "手机摄像头环",
    "target_recipe_id": "摄像头环_v2"
  },
  "current_recipe": {
    "recipe_id": "摄像头环_v1",
    "fixture_model": "夹具_12",
    "nozzle_model": "吸嘴_A3",
    "vision_profile": "视觉参数_17"
  },
  "target_recipe": {
    "recipe_id": "摄像头环_v2",
    "fixture_model": "夹具_12",
    "nozzle_model": "吸嘴_B1",
    "vision_profile": "视觉参数_21"
  },
  "diffs": [
    {
      "field": "吸嘴型号",
      "from": "吸嘴_A3",
      "to": "吸嘴_B1",
      "risk_hint": "中"
    },
    {
      "field": "视觉参数",
      "from": "视觉参数_17",
      "to": "视觉参数_21",
      "risk_hint": "高"
    }
  ],
  "station_snapshot": {
    "fixture_installed": "夹具_12",
    "nozzle_installed": "吸嘴_A3",
    "camera_profile_loaded": "视觉参数_17",
    "station_ready": true
  },
  "historical_cases": [
    {
      "case_id": "chg_1182",
      "similarity": 0.91,
      "diff_pattern": ["视觉参数", "吸嘴型号"],
      "issue": "首件偏移过高",
      "resolution": "人工确认 + 首件复核",
      "result": "调整后通过"
    }
  ],
  "rules": {
    "must_confirm_if": [
      "视觉参数发生变化",
      "引入新的吸嘴型号"
    ],
    "auto_release_if": [
      "夹具型号一致",
      "不存在高风险差异"
    ]
  }
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `station.station_id` | 当前工位唯一编号。 |
| `station.station_type` | 当前工位类型，用来匹配适用的规则和历史案例。 |
| `station.current_state` | 当前工位状态，例如是否处于可进入换型准备的阶段。 |
| `work_order.order_id` | 当前工单编号。 |
| `work_order.product_family` | 当前工单对应的产品族。 |
| `work_order.target_recipe_id` | 工单要求切换到的目标配方编号。 |
| `current_recipe.*` | 当前工位实际已加载的配方关键字段，用来和目标配方做对比。 |
| `target_recipe.*` | 本次换型希望切换到的目标配方关键字段。 |
| `diffs` | 系统预先算出的差异列表，避免把原始对象全文交给模型自己找不同。 |
| `diffs.field` | 发生变化的字段名。 |
| `diffs.from` / `diffs.to` | 该字段的当前值和目标值。 |
| `diffs.risk_hint` | 规则层预先给出的风险提示，帮助模型快速聚焦重点。 |
| `station_snapshot` | 当前现场实际状态快照，用来判断“目标配置”和“现场状态”是否一致。 |
| `station_snapshot.fixture_installed` | 当前实际装上的夹具型号。 |
| `station_snapshot.nozzle_installed` | 当前实际装上的吸嘴或末端工具型号。 |
| `station_snapshot.camera_profile_loaded` | 当前视觉系统实际加载的参数配方。 |
| `station_snapshot.station_ready` | 工位是否处于 ready，可否继续换型准备。 |
| `historical_cases` | 相似历史案例列表，通常来自检索结果，而不是全量历史数据。 |
| `historical_cases.similarity` | 当前换型与历史案例的相似度，用来排序参考价值。 |
| `rules.must_confirm_if` | 明确规定哪些条件出现时必须人工确认。 |
| `rules.auto_release_if` | 明确规定哪些条件满足时可以自动放行。 |

这里关键不在字段多少，而在输入已经被系统整理成“可比较、可引用、可审计”的结构。

### 5. 系统提示词通常应该长什么样

这里的系统提示词最好写得硬一点，不要写成通用助手风格，也不要鼓励模型自由发挥。用全中文没问题，反而更适合跟现场术语对齐。比如：

```text
你是 3C 固定工位的换型准备助手。

你的任务是根据输入的配方差异、当前工位状态、历史案例和既有规则，
输出结构化的换型风险判断、补充检查项、首件关注点和人工确认建议。

你不是主协调单元，不负责最终放行，也不负责修改 PLC、机器人程序或现场参数。

请遵守以下约束：
1. 只能依据输入数据判断，不要编造不存在的现场状态。
2. 输出必须是 JSON，不要输出解释性散文。
3. 风险等级只能使用 低、中、高。
4. 如果信息不足，必须明确标记 insufficient_information=true。
5. 不要输出底层控制指令，不要建议直接修改运动轨迹或控制参数。
6. 对补充检查项的表述要具体，可直接给工程师或班组长查看。
7. 如果规则明确要求人工确认，不要建议自动放行。
```

### 6. 用户提示词通常怎么写

用户提示词不用追求花哨，直接说明要产出什么，再把上下文 JSON 放进去就够了。例如：

```text
请基于下面的换型上下文，输出一个结构化建议包，字段必须包含：
- risk_level
- top_risks
- required_checks
- first_article_focus
- requires_human_confirm
- release_recommendation
- reasons
- insufficient_information

换型上下文如下：
{这里放上一步组装好的 JSON}
```

### 7. 最终输出什么，结构最好长什么样

模型输出最好也是固定结构，而不是自由文本。更适合接到站控或 `HMI` 的结果，可以长这样：

```json
{
  "risk_level": "高",
  "top_risks": [
    "视觉参数已变化，但现场仍未加载目标参数",
    "当前吸嘴型号与目标配方不一致"
  ],
  "required_checks": [
    "确认已安装吸嘴_B1",
    "加载视觉参数_21",
    "执行首件偏移检测"
  ],
  "first_article_focus": [
    "摄像头环对位偏移",
    "吸附取料稳定性"
  ],
  "requires_human_confirm": true,
  "release_recommendation": "确认前暂停放行",
  "reasons": [
    "相似历史案例中，此类差异通常需要首件确认",
    "当前现场状态与目标配方不一致"
  ],
  "insufficient_information": false
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `risk_level` | 当前换型的总体风险等级。 |
| `top_risks` | 当前最主要的风险点，供 `HMI`、工程师和规则层快速查看。 |
| `required_checks` | 本次换型必须补做或确认的检查项。 |
| `first_article_focus` | 首件验证阶段应重点关注的项目。 |
| `requires_human_confirm` | 是否建议进入人工确认流程。 |
| `release_recommendation` | 模型给出的放行建议，例如先挂起、待确认后放行。 |
| `reasons` | 给出上述判断的主要依据，便于审计和复盘。 |
| `insufficient_information` | 当前输入是否不足以支撑可靠判断。 |

再往下走一步，规则层通常会把这份建议包重新裁决成几种固定动作：

* `自动放行`
* `等待确认`
* `拒绝放行`

这样模型和现场放行边界就是分开的。

### 8. 这套链路的真实重点是什么

这套做法看起来像在写提示词，其实真正决定成败的是前后两层有没有站稳：

* 前面有没有把差异、规则和历史案例整理成结构化上下文
* 后面有没有把模型输出收束成固定字段和固定放行动作

这两层站不稳，大语言模型就很容易变成“会说很多，但接不进现场”的一层。上下文和输出结构都收得够硬，它才有机会成为一套真正在现场可用的换型助手。

## 边界与限制

这里有几个边界最好一开始就说清：

* `Agentic AI` 不负责修改 `PLC` 程序或机器人轨迹
* `Agentic AI` 不替代 `MES`、配方系统或站控放行逻辑
* “是否需要换型”主要依赖结构化规则比较，不依赖大模型猜测
* 人工确认只在高风险和规则要求时介入，不应默认常开
