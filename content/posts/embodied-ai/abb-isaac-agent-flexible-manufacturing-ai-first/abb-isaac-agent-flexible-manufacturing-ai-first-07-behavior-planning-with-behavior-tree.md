# 第 7 章补篇：为什么行为规划更适合用行为树

摘要：这一篇不再泛谈“行为树很适合机器人”，而是固定用一次跨站补料与空箱回收任务，把流程图、状态机和行为树放在一起比较。重点不是谁更高级，而是谁更适合承接任务级行为规划。

## 先固定同一个任务

后面都围绕同一个任务：

```text
当前任务：从 A 仓位取一箱连接器，送到 3 号装配站，并回收空箱
执行中异常：去往 3 号站途中发现该站暂时占用，机器人剩余电量降到 18%
系统候选：先去临时缓存位 -> 判断是否需要补能 -> 去充电点补能 -> 重新规划路线 -> 到站交付 -> 回收空箱
若仍无法完成：转人工调度
```

这个例子刚好把行为规划最难处理的几件事都拉出来了：

* 站点占用会变
* 电量会继续下降
* 恢复路径不是整条任务重来，而是插入缓存、补能和重规划
* 同一套恢复骨架还希望复用到别的站点和别的补料任务

所以这里真正要解决的，不是“怎么把步骤列出来”，而是“怎么把这条任务链组织成一套能执行、能回退、能复用的结构”。

## 为什么这里需要行为规划

如果任务永远是线性的，普通流程说明就够了。但本文这个任务不是这样。

系统需要持续判断：

* 3 号站现在能不能交付
* 缓存位是不是空着
* 当前电量够不够继续等
* 等待之后是继续交付、先补能，还是转人工

这意味着规划结果不能只是一串动作列表。它必须把：

* 条件检查
* 主路径
* 恢复路径
* 人工接管点

放在同一个执行骨架里。

这就是本文说的“行为规划”。

## 流程图、状态机、行为树分别在回答什么

这三者都能描述这个场景，但回答的问题不同。

流程图回答的是：这条任务大概怎么走。

* 它适合说明从取料、去站点、去缓存位、补能、重规划到最终交付的处理顺序。
* 它最适合文档说明、沟通和讨论业务流程。
* 它的问题是，一旦继续叠加等待、重试、补能插入和多条恢复支路，图会迅速变长。

状态机回答的是：系统会有哪些状态，状态之间怎样转移。

* 它适合做设备级和模块级控制，比如导航状态、充电状态、站点握手状态。
* 它能把“去缓存位中”“等待站点释放中”“充电中”“重规划中”写得很明确。
* 它的问题是，站点占用、电量阈值、路线禁行、缓存位可用性这些本来只是条件的东西，很快会被抬成状态，最后变成状态膨胀。

行为树回答的是：这条任务闭环怎么执行、怎么回退、怎么复用。

* 它适合把条件节点、动作节点和恢复分支放在同一棵树里。
* 它天然支持局部插入子流程，比如站点占用时插入缓存等待，电量不足时插入补能。
* 它最适合任务级执行编排。

## 为什么这个场景更适合行为树

这个任务里，行为树真正占优的不是“更先进”，而是更贴合问题本身。

第一，它把条件和状态拆开了。

* “3 号站是否可对接”
* “当前电量是否足够”
* “缓存位是否可用”

这些都可以是条件节点，不必硬塞进状态定义。

第二，它支持局部恢复。

这条任务失败时，通常不是整条任务报废，而是插入一个局部子树：

* 去缓存位
* 去充电点
* 重新规划
* 再次尝试交付

这比整条任务从头再跑合理得多。

第三，它支持复用。

检查电量、去缓存位、执行补能、重新规划、到站交付、回收空箱，这些子树不只服务 3 号站，也可以服务别的站点和别的补料链。

第四，它天然贴近运行时。

行为树不是一张说明图，而是一棵可执行树。运行时可以持续 tick，依据 `success / failure / running` 推进下一步。

## 如果把这个任务写成行为树

把这个场景收成一棵树，大致会是这样：

```text
Fallback
├── Sequence
│   ├── 检查料箱已装载
│   ├── Fallback
│   │   ├── Sequence
│   │   │   ├── 检查 3 号站可对接
│   │   │   ├── 检查电量足够
│   │   │   ├── 去 3 号站
│   │   │   ├── 交付料箱
│   │   │   └── 回收空箱
│   │   └── Sequence
│   │       ├── 去临时缓存位
│   │       ├── 如有需要则去充电点补能
│   │       ├── 重新规划路线
│   │       ├── 再次检查 3 号站可对接
│   │       ├── 去 3 号站
│   │       ├── 交付料箱
│   │       └── 回收空箱
└── 人工调度
```

这棵树的关键不是“长得像树”，而是它把这件事拆成了三类节点：

* 条件节点：检查站点、电量、缓存位
* 动作节点：导航、补能、交付、回收
* 结构节点：`Sequence` 和 `Fallback`

这样一来，恢复路径就不再是事后补丁，而是任务骨架的一部分。

## 三种表达方式在工业里的实际分工

这三者不是互斥替代关系，更实际的分工通常是：

* 流程图负责讲清楚任务全貌和系统边界
* 状态机负责导航、充电、站点握手这类设备级确定性控制
* 行为树负责任务级行为规划、条件判断和恢复编排

所以这里真正的结论不是“以后都用行为树”，而是：

* 讲清楚流程，用流程图
* 写底层控制，用状态机
* 做任务级行为规划，用行为树

对“跨站补料与空箱回收”这个任务，行为树正好卡在最合适的位置。
