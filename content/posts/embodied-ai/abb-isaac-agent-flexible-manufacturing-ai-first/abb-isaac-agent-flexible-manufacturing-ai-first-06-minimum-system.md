# 第 6 章：Agentic AI 在 3C 柔性制造中的产品位置与最小形态

摘要：这一章先回答 Agentic AI 产品在传统 3C 制造里处在什么位置，再回答它的能力底座应该是什么。对生产场景来说，这个产品本身应围绕 `LLM`、世界模型和空间智能提供柔性制造能力，同时把节拍、确定性和工业边界放在前面。后面再分成两类场景展开：固定工位的 3C 组装 / 分拣，以及可移动机械臂智能。

## 名词解释

| 名词 | 说明 |
| --- | --- |
| `HMI` | Human Machine Interface，人机界面，用于现场查看状态和执行操作。 |
| `上位机` | 位于设备控制之上的工业计算机或软件系统，负责监控、调度与交互。 |
| `能力闭环` | 从输入、决策、执行到评估与回放完整连通的验证结构。 |
| `Agentic AI` | 以大模型为核心，能够理解任务、调用工具、做阶段性决策并与外部系统协同的产品形态。 |
| `回放` | 基于历史日志重现任务过程，用于分析、评估和定位问题。 |
| `世界模型` | 更接近一种对状态演化、动作后果和下一步轨迹做预测或推演的模型能力。在工业场景里，常被落到恢复路径评估、状态演进判断和回放分析上。 |
| `空间智能` | 对空间位置、姿态、约束关系和物体交互的建模能力，常落在视觉理解、位姿估计、接触判断和空间关系推断上。 |
| `节拍` | 产线或工站完成一个作业循环所允许的时间窗口。 |
| `技能接口` | AI 系统调用执行能力时使用的标准输入输出接口。 |
| `约束适配层` | 把 AI 决策转换为符合工业边界的可执行请求与限制条件的中间层。 |
| `ROS 2` | 当前机器人与柔性制造研发中常见的节点通信与算法集成框架，适合承载视觉、力控、状态聚合、仿真和算法模块之间的通信。 |
| `DDS` | `ROS 2` 常用的底层通信中间件，负责节点发现、发布订阅、序列化和传输。 |
| `OPC UA` | 工业现场常见的标准化通信协议，适合从 `PLC` 或站级网关读取结构化状态与设备信号。 |
| `ABB PC SDK / Robot Web Services` | `ABB` 控制器常见的上位接入方式。`PC SDK` 适合开发 PC 侧控制应用；`Robot Web Services` 适合通过 `HTTP REST` 访问控制器资源。 |
| `REST / gRPC` | 常见服务接口方式，适合智能体服务、视觉服务、站级编排服务之间交换结构化请求和结果。 |
| `WebSocket` | 适合把工站实时状态、建议结果和人工确认事件推送到 `HMI` 或监控界面。 |

前一章讲的是试点怎么定义，这一章讨论的是产品应该站在什么位置，以及这个产品最小要长成什么样。这里不讨论实验室里的临时拼接，也不讨论大而全的平台。重点有三件事：

* `Agentic AI` 产品在传统工业 3C 制造里处在什么位置
* 这个产品应围绕哪些能力底座建立
* 在节拍和工业边界约束下，固定工位与可移动机械臂两个场景分别应怎样落地

## 1. Agentic AI 产品在传统工业 3C 制造里的位置

### 1.1 它不是 PLC、机器人控制器或 MES 的替代品

在传统 3C 制造里，`PLC`、机器人控制器、工艺设备和 `MES` 各自都有明确职责。`Agentic AI` 产品不应该去替代这些系统，而应该插在“任务语义”和“工业执行”之间。

更准确地说，它的位置是：

* 上接 `HMI`、`MES`、`SOP`、配方、工艺规则
* 横向连接视觉、力控、知识库、案例库、仿真与回放系统
* 下接编排层、`ROS 2` 节点层、`PLC / OPC UA`、`ABB` 控制器适配层

所以它不是一个“更聪明的控制器”，而是一个把任务理解、状态判断、恢复建议和技能编排接进工业系统的产品层。

### 1.2 这个产品的能力底座应围绕 LLM、世界模型和空间智能

如果把 `Agentic AI` 产品拆开看，最核心的能力底座通常有三类：

| 能力底座 | 主要作用 | 在柔性制造里的典型产物 |
| --- | --- | --- |
| `LLM` | 理解任务、整理上下文、生成结构化建议、调用知识与工具 | 恢复建议、换型建议、任务模板、人工确认清单 |
| `世界模型` | 预测状态演化、推演动作后果、评估恢复路径 | 恢复路径评估、状态演进判断、回放与预测 |
| `空间智能` | 理解位姿、空间关系、接触状态和场景几何约束 | 位姿结果、补偿量、接触判断、空间约束检查 |

这三类能力不是平行摆设。

* `LLM` 负责把多源信息组织成可用决策
* 世界模型负责让系统预测“当前状态接下来会怎样演化，这个动作之后大概率会发生什么”
* 空间智能负责把视觉、位姿、接触和空间关系接到具体工艺动作上

没有 `LLM`，系统很难把复杂上下文整理成任务判断。没有世界模型，系统就更难对状态演化和动作后果做推演，只能更被动地对当前帧做反应。没有空间智能，系统很难把柔性制造里的变化真正落到抓取、对位、插入、分拣和移动操作上。

这些图最关键的不是“画了多少层”，而是把主生产回路和旁路智能层的关系讲清楚：

* 主生产回路里通常存在一个主协调单元。它不是单指某个概念层，而是一个落在真实硬件上的控制角色，常见形态是 `PLC` 硬件 + `PLC` 程序，或者站控工控机 + 站控软件。
* `ROS 2` 确定性节点、`ABB` 控制器和现场设备属于执行侧，按主协调单元给出的节奏和资格去完成动作。
* `Agentic AI` 默认站在回路外侧，读取结构化状态、异常事件和历史片段，不直接进入毫秒级控制闭环。
* `Agentic AI` 可以影响现场，但影响方式应受限，通常只有三类：恢复建议、换型准备增强、受限技能候选。
* 换型准备不是 `Agentic AI` 带来的新流程，而是传统产线本来就存在的准备逻辑。常见承载方式是 `MES` 的工单 / 配方切换状态、`HMI` 检查页、站控放行条件和现场 `SOP`。`Agentic AI` 的作用，是把这套准备逻辑做得更智能、更清晰、更少依赖个人经验。
* `世界模型` 和空间智能都属于旁路智能层内部能力，它们帮助系统理解状态和空间约束，但它们不替代控制器、`PLC` 或工艺互锁。
* 世界模型对应的也是一类真实痛点场景。传统自动化当然能做异常分支，但更常见的做法是把很多情况都压成少数几个报警码，再靠 `if-else`、重试次数和人工经验去处理。它真正不擅长的，是识别“当前到底处于哪一种失败状态”，以及“下一步最合理的恢复路径是什么”。
* 这正是世界模型要补的空白：把“抓取失败”“抓到了但姿态不稳”“已到对位位但扣合未到位”“连续失败不应再自动恢复”这类状态阶段区分开，再把恢复路径选择做得比纯规则更细。
* `MES`、`WMS`、`QMS` 提供的是任务上下文，`HMI` 和人工确认提供的是人机交互入口，它们都不替代主生产回路。

从供给关系看，这套东西通常也是多方拼出来的：

* 主生产回路大多由自动化团队、机器人厂商和集成商负责，重点是确定性、互锁和节拍。
* `Agentic AI` 应用、世界模型、空间智能、建议逻辑和回放分析，通常才是产品团队最核心的自研部分。
* `MES`、`WMS`、`QMS` 往往是企业现有系统，产品团队更多是在做对接，而不是重建。

### 1.4 节拍和实效性决定了这个产品不能站错位置

工业现场不会为了引入 `Agentic AI` 就放弃节拍。

这意味着一个很重要的边界：毫秒级连续控制、插补、伺服和硬实时互锁，仍然应该留在机器人控制器、`PLC` 和确定性的 `ROS 2` 节点里。`Agentic AI` 产品更适合处理这些时间尺度上的工作：

* 在读完一轮状态输入后的 `0.1` 秒到几秒内，判断当前处于什么状态，并选择下一步调用哪个已验证技能
* 一个工步内的恢复建议和异常分流
* 一个循环内的补偿量判断、路径选择和人工确认
* 换型、配方切换、任务模板选择这类秒级到分钟级问题

比如在固定工位装配里，系统读到视觉偏差、吸附状态和机器人回执后，要在很短时间内判断“这次是继续对位、先重拍一张、低速重抓一次，还是直接转人工复位”。这就是这里说的状态理解和技能选择。

这里还有一个容易被误解的点：`Agentic AI` 不等于每次在线决策都直接调用大语言模型。对有节拍要求的单站，更现实的做法通常是规则、缓存、状态机和本地小模型优先，复杂情况再升级到 `LLM`。否则把每一轮异常分流都压到远程大语言模型上，在线耗时通常很难稳定控制在 `150ms - 300ms`。

换句话说，它应该站在“决定调用哪个技能、何时转人工、是否需要恢复、如何解释当前状态”这一层，而不是直接进入毫秒级控制回路。

## 2. 两类场景决定了产品形态不完全一样

### 2.1 固定工位的 3C 组装 / 分拣，是更近的主战场

对大多数传统工业 3C 团队来说，更近、更容易形成产品的是固定工位场景，比如：

* 组装单站
* 分拣单站
* 贴附、对位、扣合、复检单站
* 多配方切换但站内边界稳定的柔性工位

这类场景的特点是：

* 机器人、夹具、相机、工艺设备位置基本固定
* `PLC`、工艺互锁和节拍边界清楚
* `Agentic AI` 主要解决的是换型、异常恢复、补偿、技能选择和人工协同

因此，这类场景里的产品重点通常是：

* 任务状态理解
* 配方和工艺上下文组织
* 恢复建议
* 补偿与分流
* 人工确认与回放评估

这也是当前更适合做生产最小产品形态的主战场。

### 2.2 可移动机械臂智能，对世界模型和空间智能要求更高

第二类场景是可移动的机械臂智能，比如：

* 移动底盘 + 机械臂的巡检 / 搬运 / 上下料
* 半结构化车间里的移动作业单元
* 需要在多个工位之间切换任务的移动作业系统

这类场景和固定工位最大的区别不在机械臂，而在环境本身更开放：

* 场景边界更弱
* 位姿关系更动态
* 路径和作业点不再完全固定
* 世界模型和空间智能的重要性显著上升

因此，这类产品除了 `LLM` 任务理解之外，还要更依赖：

* 环境建图和定位
* 多视角空间感知
* 场景状态维护
* 任务切换和路径级恢复

它依然需要 `ROS 2`、`PLC`、控制器接入和日志回放，但产品重心会从“站内任务理解”进一步扩展到“场景理解和跨位置执行”。

如果要把这一类场景具体化，可以参考这张图：

* 半结构化工厂里的跨工位移动操作：[abb-isaac-agent-flexible-manufacturing-ai-first-06-mobile-manipulation-cross-station.puml](./diagrams/abb-isaac-agent-flexible-manufacturing-ai-first-06-mobile-manipulation-cross-station.puml)

### 2.3 两类场景的共同点和差异

| 维度 | 固定工位 3C 组装 / 分拣 | 可移动机械臂智能 |
| --- | --- | --- |
| 主战场 | 站内柔性、换型、恢复、补偿 | 场景变化、任务切换、跨位置执行 |
| 主要约束 | 节拍、互锁、工艺窗口、设备资格 | 定位、路径、场景变化、任务切换 |
| 更依赖的能力 | `LLM` + 工艺上下文 + 状态机 | `LLM` + 世界模型 + 空间智能 |
| 更典型的工业栈 | `PLC / OPC UA`、`ABB` 控制器、视觉与工艺设备 | `ROS 2`、导航 / 感知节点、机械臂控制器 |
| 更适合先形成产品的场景 | 是 | 通常不是第一步 |

## 3. 生产最小产品分层

### 3.1 分层的起点不是设备，而是产品能力

从产品角度看，这套系统不能只按设备分层，而要按能力分层。这样做的好处是，固定工位和可移动机械臂两类场景可以共用同一套产品骨架，只是在下层运行时和执行侧有所不同。

## 4. 主流技术栈下的最小产品形态

### 4.1 固定工位 3C 组装 / 分拣的推荐形态

如果目标是一个最先能落地的产品，固定工位场景更推荐采用下面这套主流栈：

| 层级 | 主要模块 | 推荐技术 |
| --- | --- | --- |
| 产品入口层 | `HMI`、工单、配方、人工确认 | Web / 上位机界面 |
| Agentic AI 能力层 | `LLM`、案例检索、状态理解、恢复建议 | 智能体服务、知识库 |
| 编排与约束层 | 任务状态机、资格裁决、节拍控制 | 编排服务 |
| 工业运行时层 | 视觉、力控、状态节点、互锁、控制器接入 | `ROS 2`、`PLC / OPC UA`、`ABB PC SDK / Robot Web Services` |
| 执行层 | 固定工位机器人、夹具、相机、工艺设备 | `IRB 1100`、视觉系统、工艺设备 |

这类产品里，`Agentic AI` 最典型的落点是：

* 换型建议
* 补偿与复检决策
* 异常恢复
* 人工确认分流
* 多技能候选排序

### 4.2 可移动机械臂智能的推荐形态

如果目标是可移动机械臂智能，骨架仍然类似，但运行时重点会变化：

| 层级 | 固定工位重点 | 可移动机械臂重点 |
| --- | --- | --- |
| Agentic AI 能力层 | 工艺上下文、恢复建议、换型 | 场景理解、任务切换、跨位置决策 |
| 编排与约束层 | 站级资格、工艺边界、节拍控制 | 位置状态、任务切换、路径级边界 |
| 工业运行时层 | `PLC / OPC UA`、视觉、`ABB` 接入 | `ROS 2`、定位 / 建图 / 感知节点、机械臂控制器 |
| 执行层 | 固定工位机器人与设备 | 移动底盘 + 机械臂 + 场景感知 |

这类场景依然可以用同一套产品骨架，但世界模型和空间智能的权重会明显更高。

### 4.3 节拍和时间尺度应该怎样分工

不管是固定工位还是可移动机械臂，时间尺度都要先分清：

| 时间尺度 | 典型任务 | 适合放在哪一层 |
| --- | --- | --- |
| 毫秒级 | 伺服、插补、硬实时互锁 | 机器人控制器、`PLC`、确定性节点 |
| 10-100 毫秒 | 视觉节点、力控节点、状态聚合 | `ROS 2` 运行时 |
| 100 毫秒到秒级 | 技能选择、恢复判断、人工确认分流 | 编排与约束层、Agentic AI 能力层 |
| 秒级到分钟级 | 换型、配方切换、任务模板调整 | `LLM`、世界模型、产品入口层 |

把这个边界先讲清，后面很多争论会自然消失。  
`Agentic AI` 产品不是为了进入毫秒级控制回路，而是为了把更上层的柔性能力落进工业系统。

### 4.4 一个真实可交付的最小部署清单

如果只保留真实交付最需要的模块，这个最小产品形态至少应包含：

* `HMI`
* 智能体服务
* 编排与约束层
* `ROS 2` 运行层
* `PLC / OPC UA`
* `ABB` 控制器适配层
* 视觉服务
* 日志 / 回放 / 评估

这不是为了把系统做大，而是为了保证：

* 产品能接进现场
* 建议能穿过工业边界
* 执行结果可追
* 后续版本可比较

## 5. 首个版本哪些接口必须第一天就定义

下面这一节的接口、状态和样例，默认都以固定工位的 3C 组装 / 分拣单站为例。可移动机械臂场景也会用到同样的骨架，但运行时节点、状态项和空间约束会更复杂。

### 5.1 状态接口必须先于模型接口

最容易被低估、但最该最早定义的，不是模型 API，而是状态接口。

至少应明确：

* 当前任务阶段如何表示
* 当前产品上下文如何表示
* 当前异常类型如何表示
* 当前技能状态如何表示
* 当前人工接管点如何表示

这些不先定，后面几乎所有模块都会返工。

放到一个手机配件安装单站里，这件事往往比模型选型更早决定成败。比如同样是“没有完成抓取”，系统到底记成 `pick_failed`、`vacuum_not_built`、`pose_invalid` 还是 `fixture_not_ready`，会直接决定后面恢复建议、案例检索和技能路由还能不能成立。如果前期只是笼统地记成“抓取异常”，等到要比较“先重拍还是先重抓”时，日志就已经不够用了。

### 5.2 决策输出接口必须天然可审计

决策输出不能只是一句自然语言建议，或者一个隐式动作。

更合理的输出形式应至少包含：

* 决策类型
* 决策依据摘要
* 推荐技能或推荐路径
* 前置条件
* 是否需要人工确认
* 失败后的默认回退方式

不然你后面很难做：

* 责任分析
* 结果对比
* 失败复盘

在 3C 单站里，一个可审计的决策输出应该像“当前判断为吸附失败后的恢复决策，推荐 `vision_retake`，前置条件是相机 ready 且安全门关闭，不需要人工确认，失败后回退到 `manual_confirm`”，而不是一句“建议重新试一次”。把决策写成这种结构，现场工程师、运维和后续评估系统才有共同语言。

### 5.3 技能接口必须能回执、能超时、能失败

技能接口不能只定义：

* 调用了哪个能力

还必须定义：

* 前置条件是否满足
* 执行是否成功
* 失败属于哪一类
* 超时后如何处理

少了这些，任务层再聪明也很难形成稳定闭环。

例如，站级编排服务调用一次 `pick_retry_low_speed` 技能后，返回的不应只有“成功/失败”。更实用的回执应该至少包含：前置条件是否满足、是否真的发起执行、执行超时没有、失败是吸附失败还是路径中断、是否已经退回安全位。否则智能体只知道“又失败了”，却不知道下次该换策略、转人工，还是继续同一路径。

### 5.4 日志接口必须和状态、决策、技能接口一起设计

很多团队不是没有日志，而是日志补得太晚，最后很难和任务语义对齐。

更合理的做法是第一天就一起定义：

* 状态怎么记
* 决策怎么记
* 技能调用怎么记
* 人工接管怎么记
* 最终结果怎么打标签

这直接决定后面系统能不能沉淀成数据资产。

如果放到手机配件安装试点里，日志最好能把一次完整闭环串起来：某个版本在什么阶段发生了什么异常，系统推荐了哪条恢复路径，现场是否接受了这条建议，技能执行结果怎样，最后是成功安装、人工接管，还是停机退出。只有这种成链条的日志，后面才可能真的做回放、对比分析和经验沉淀；如果日志只是零散的错误码和时间戳，系统就很难持续改进。

### 5.5 一个可直接落地的接口草案

如果把试点具体落到“手机配件安装流程智能体”，首版接口不要从自然语言开始，先把上下文、决策和技能请求做成结构化对象。

一个最小的上下文输入可以长这样：

```json
{
  "station_id": "3c_cell_01",
  "product_version": "camera_ring_v2",
  "task_stage": "pick_ready",
  "plc_state": {
    "fixture_ready": true,
    "vacuum_ok": false,
    "camera_ready": true,
    "safety_door_closed": true
  },
  "vision_result": {
    "part_found": true,
    "pose_ok": false,
    "offset_mm": 0.42
  },
  "last_errors": [
    "vacuum_fail",
    "vacuum_fail"
  ],
  "operator_mode": "auto"
}
```

一个最小的决策输出可以长这样：

```json
{
  "decision_type": "recovery",
  "recommended_skill": "vision_retake",
  "reason": "连续两次吸附失败，当前视觉可用，但姿态不稳定",
  "preconditions": [
    "camera_ready == true",
    "safety_door_closed == true"
  ],
  "requires_human_confirm": false,
  "fallback": "manual_confirm"
}
```

技能请求则不应是“帮我重新抓一下”这种模糊表达，而应是受限、可审计的标准报文。下面示例里的 `recipe_id` 指的是当前要使用的视觉参数配方标识：

```json
{
  "skill_name": "pick_retry_low_speed",
  "request_id": "req_20260331_0001",
  "params": {
    "recipe_id": "camera_ring_v2",
    "speed_profile": "low",
    "retry_index": 2
  },
  "timeout_ms": 5000,
  "on_timeout": "return_safe_pose",
  "requires_human_confirm": false
}
```

用伪代码表达这套逻辑，站级编排服务至少要做下面这几步：

```python
def handle_agent_decision(context, decision):
    if not context["plc_state"]["safety_door_closed"]:
        return reject("safety_not_ready")

    if decision["recommended_skill"] == "pick_retry_low_speed":
        if not context["plc_state"]["fixture_ready"]:
            return reject("fixture_not_ready")
        return call_skill(
            skill_name="pick_retry_low_speed",
            params={
                "recipe_id": context["product_version"],
                "retry_index": len(context["last_errors"]),
            },
        )

    if decision["requires_human_confirm"]:
        return push_hmi_confirm(decision)

    return reject("unsupported_decision")
```

这段伪代码要表达的重点不在语言本身，而在结构：

* 智能体先输出结构化建议
* 站级编排服务检查资格条件
* 满足条件才调用技能
* 不满足条件就拒绝、回退或转人工确认

这才像一个能在 `ABB` 3C 单站里真正落地的最小接口闭环。

如果要把这条链路落实到团队分工和接口实现上，可以直接按下面这张总表拆：

| 链路阶段 | 谁提供 | 典型内容 | 推荐协议/形式 | 关键要求 |
| --- | --- | --- | --- | --- |
| 状态输入 | `PLC / IO` 网关、视觉服务、工艺设备、人工确认端 | 产品版本、任务阶段、设备 ready、异常码、视觉偏差、人工接管状态 | `OPC UA`、`REST / gRPC`、`WebSocket` 事件 | 必须结构化、可追时间戳、能区分真实信号和上层推断 |
| 决策输出 | 智能体服务 | 当前判断、推荐技能、前置条件、是否需人工确认、默认回退策略 | `JSON` / `gRPC` 响应 | 必须可审计，不能只返回自然语言 |
| 技能请求 | 站级编排服务 | `vision_retake`、`align_to_pose`、`pick_retry_low_speed`、`return_safe_pose` 等标准请求 | 内部 `REST / gRPC`、执行适配层接口 | 只能调用已验证技能，不能直接下底层连续控制命令 |
| 技能回执 | `ABB` 执行适配层、视觉服务、站级技能服务 | 是否执行、执行结果、失败类型、超时、是否已回退 | `JSON` / 结构化回执 | 必须返回失败原因，不能只有 success / fail |
| 日志记录 | 站级编排服务统一落库 | 状态快照、决策记录、技能调用、人工确认、最终标签 | 事件日志、数据库记录、回放文件 | 必须能把一次完整闭环串起来，支持回放和对比分析 |

这五类接口里，只要有一类没定清，所谓“智能体系统”就很容易退化成一次性集成，而不是能继续演进的产品底盘。

## 6. 哪些东西不应该成为首版系统的核心

### 6.1 不要先做“大一统 Agent 平台”

首版系统很容易在这里浪费时间：先做一个

* 能接多模型
* 能配多工具
* 能支持多场景
* 能扩展多工站

的平台。

这类工作后面也许有价值，但放在第一版，通常只会稀释重点。

### 6.2 不要让 UI 先于能力闭环

另一个常见偏差，是过早追求：

* 漂亮的前端
* 复杂的监控面板
* 丰富的操作入口

这些都可以有，但别排在下面这些东西前面：

* 状态闭环
* 决策闭环
* 回退闭环
* 评估闭环

### 6.3 不要把连续动作控制当成第一版主战场

除非试点本身就明确验证连续控制相关能力，否则第一版更不应把主要精力放在：

* 端到端动作输出
* 连续低层控制
* 复杂闭环轨迹学习

这些方向当然有价值，但如果把它们当成首版产品的主战场，不确定性会放大得很快。

## 结论

生产可落地的最小产品形态，不是把控制器、服务和模型堆在一起，而是把单一能力闭环接稳。  
真正要盯住的，是上下文与状态是否够用，`ROS 2` 有没有把视觉与算法节点接稳，`PLC / OPC UA` 有没有把站级资格读清，`ABB PC SDK / Robot Web Services` 有没有把控制器接好，决策有没有穿过约束适配层，日志和回放有没有真正串起来。把这几件事做好，系统就有落地价值；做不好，再复杂也更像演示。
