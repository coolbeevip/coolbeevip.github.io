# AI-first 柔性制造探索，第 6 章：异常分流链路

摘要：异常分流是固定工位里比较容易做出产品价值的旁路智能场景。重点不在于让 `Agentic AI` 进入毫秒级控制闭环，而是在主协调单元收到异常后，尽快给出恢复建议、分流建议和需要时的人工确认建议。

## 场景定义

在 3C 小型精密装配单站里，异常并不稀少。常见情况包括：

* 真空吸附失败
* 视觉偏差超限
* 抓取成功但姿态不稳
* 对位失败
* 连续重试后仍未恢复

传统自动化当然能处理异常，但现场更常见的做法，是把很多失败压成少数几个报警码，再配上固定重试次数，剩下的交给工程师经验。`Agentic AI` 在这里补上的，是把异常拆得更细，再从里面挑一条更合适的恢复路径。

## 链路拆解

### 1. 主生产回路先接住异常

异常首先由执行侧返回给主协调单元。常见链路是：

* `ABB` 控制器和现场设备返回异常事件与状态快照
* 主协调单元先记录事件，再把当前状态送进异常分流链路
* 系统在有限时间窗口内拿到恢复建议
* 是否继续恢复、继续重试还是转人工，仍由主协调单元裁决

这一链路里，旁路智能可以参与判断，但不会绕开主协调单元直接下发动作。

### 2. 传统异常处理为什么不够用

很多固定工位在没有 `Agentic AI` 的情况下，也能把异常处理链路跑起来。常见做法是：

* 控制器或 `PLC` 返回报警码
* 站控按报警码进入固定分支
* 超过重试次数后转人工

这套做法一碰到稍复杂一点的现场情况，就会显得过粗：

* 不同异常被压成同一类报警，系统分不清差异
* 恢复动作比较单一，通常就是重试几次或者直接转人工
* 人工接手时上下文不完整，只能临场判断
* 后续复盘很难回答，到底是异常识别错了，还是恢复路径选错了

### 3. 结构化状态送入旁路智能层

送进旁路智能层的输入，应该是结构化状态，例如：

* 当前工步
* 当前料号或配方
* 视觉偏差量
* 吸附状态
* 最近几次失败类型
* 当前是否允许自动恢复

这些字段定不清，后面的建议也很难稳定下来。

### 4. 历史案例与当前状态一起参与建议生成

`Agentic AI` 可以结合共享日志和历史案例，判断当前异常更接近哪类已知情况，再给出建议，例如：

* 建议重拍一张
* 建议低速重抓一次
* 建议直接回安全位
* 建议转人工复位

### 5. 必要时再走人工确认

如果建议路径本身带风险，或者规则明确要求人工确认，就把解释和建议同步给 `HMI`。典型情况包括：

* 已连续失败多次
* 当前恢复动作接近次数上限
* 这次动作可能影响良率或设备安全边界

人工确认完成后，主协调单元再决定是否执行。

## 时间预算为什么是这个场景的关键

这个场景虽然不是毫秒级控制，但也不是可以慢慢想的离线分析。只要它介入单站节拍，就得尊重现场时间预算。

现场通常会按下面这个量级卡预算：

* 主协调单元裁决目标约 `10ms - 30ms`
* 历史读取目标约 `10ms - 30ms`
* 规则命中或缓存命中，建议生成约 `10ms - 50ms`
* 本地小模型参与判断，建议生成约 `50ms - 200ms`
* 如果要走远程大语言模型，在线耗时通常会落到 `300ms - 1500ms`，波动也更大

所以整条异常分流链路如果想稳定压在 `150ms - 300ms`，一般要靠规则、缓存和本地小模型优先。远程大语言模型更适合处理复杂长尾情况，或者退到人工确认前的建议层，不适合默认承担每一次在线裁决。

这里更像“有限预算内的同步裁决”，不是完全异步的旁路建议。真正更偏异步的部分，通常是日志沉淀、案例回写、回放分析和后续评估，不是当前这一拍到底要不要继续恢复。

## 这个场景里 Agentic AI 的真实位置

这个场景里，`Agentic AI` 更像一层异常理解和恢复建议服务，而不是底层控制器。它大致站在下面这个位置：

* 上接站控汇总后的结构化状态、历史案例和规则
* 横向接 `HMI`、日志和回放系统
* 下接一组已经验证过的恢复动作
* 最终是否执行哪条恢复路径，仍由主协调单元决定

它解决的不是“机器人怎么控”，而是“现在更像哪一种异常，下一步该走哪条恢复路径”。

## 这个场景里最关键的产品价值

异常分流的价值不只是“给一个建议”，它更像是在补系统原来缺的几块能力：

* 区分不同异常的真实阶段
* 让恢复路径从单一重试变成受限候选选择
* 让人工介入有上下文解释，而不是只看到报警码
* 让后续回放和评估能比较不同恢复路径的结果

这也是它适合拿来做首版场景的原因。

## 首版系统最值得做的增强点

如果这一块只做一版最小产品，通常先补下面几件事就够了：

* 先把异常状态收成固定字段，而不是只留报警码
* 先把恢复动作收成受限候选，而不是让系统自由生成动作
* 先把历史案例做成可检索的结构化记录，而不是只留文本日志
* 先把人工确认入口补齐，让班组长看到的是状态解释和建议，不只是报警号
* 先把日志串成完整闭环，后面才有回放和比较的基础

第一版先把异常识别和恢复建议做细，比一开始追求复杂智能更有用。

## 这一节如果真的要做，通常怎么落地

真正落地时，这条链路一般会拆成三层：

* 站控先把当前工步、异常码、视觉结果、最近几次失败记录整理成状态快照
* `Agentic AI` 根据状态快照和历史案例生成一份恢复建议包
* 规则层或主协调单元再把建议包裁决成固定动作，例如重拍、重抓、回安全位或转人工

这里最该避免的是两件事：

* 直接把原始报警和杂乱日志全文塞给模型
* 让模型自己生成底层动作而不经过受限技能集合

更实用的做法，是把输入和输出都收成固定结构。模型只负责识别异常阶段、比较恢复路径和解释建议，真正的执行仍走已验证技能。

### 1. 什么情况下值得调用大语言模型

不是每次异常都值得调大语言模型。更常见的做法是先用规则和状态机筛一遍：

* 如果异常类型简单、恢复路径固定，就直接走规则分支
* 如果当前状态组合复杂，或者需要在多个恢复动作之间比较，再调用大语言模型
* 如果已经超过最大自动恢复次数，或者当前状态明显要求人工介入，就直接转人工，不再让模型参与

目的很简单：把大语言模型留给那些“靠报警码已经分不细”的情况。

### 2. 一个更实用的异常状态输入通常长什么样

真正送进模型的，不应该只是一个报警码，而应该是一份整理过的状态快照。一个更实用的输入，通常会包含：

* 当前工位和当前工步
* 当前异常类型
* 视觉与执行结果
* 最近几次失败记录
* 当前允许的恢复动作
* 相似历史案例

可以长成这样：

```json
{
  "station": {
    "station_id": "3c_cell_01",
    "station_type": "排线对位装配",
    "task_stage": "抓取后对位前"
  },
  "abnormal_event": {
    "event_code": "真空吸附失败",
    "event_level": "中",
    "occurred_at": "2026-04-07T10:21:35+08:00"
  },
  "current_snapshot": {
    "vision_offset_mm": 0.42,
    "vacuum_ok": false,
    "pose_stable": false,
    "retry_count": 2,
    "station_ready": true
  },
  "allowed_recovery_actions": [
    "重拍",
    "低速重抓",
    "回安全位",
    "转人工复位"
  ],
  "recent_failures": [
    "真空吸附失败",
    "真空吸附失败"
  ],
  "historical_cases": [
    {
      "case_id": "abn_202",
      "similarity": 0.88,
      "issue": "连续吸附失败",
      "resolution": "低速重抓后恢复",
      "result": "通过"
    }
  ],
  "rules": {
    "max_auto_retry": 3,
    "must_confirm_if": [
      "连续失败次数 >= 3",
      "恢复动作接近工艺上限"
    ]
  }
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `station.station_id` | 当前工位编号。 |
| `station.station_type` | 当前工位类型，用来匹配适用的恢复模板和历史案例。 |
| `station.task_stage` | 当前工步阶段，决定异常发生在抓取后、对位前还是扣合阶段。 |
| `abnormal_event.event_code` | 当前异常事件的结构化名称。 |
| `abnormal_event.event_level` | 当前异常等级，通常来自规则层或站控层。 |
| `abnormal_event.occurred_at` | 异常发生时间，用来做追溯和日志关联。 |
| `current_snapshot.vision_offset_mm` | 当前视觉偏差量。 |
| `current_snapshot.vacuum_ok` | 当前吸附状态是否正常。 |
| `current_snapshot.pose_stable` | 当前姿态是否稳定。 |
| `current_snapshot.retry_count` | 当前已自动重试的次数。 |
| `current_snapshot.station_ready` | 工位当前是否仍允许继续恢复。 |
| `allowed_recovery_actions` | 当前规则允许调用的恢复动作集合。 |
| `recent_failures` | 最近几次失败记录，用来判断是否连续落在同一类异常上。 |
| `historical_cases` | 相似历史案例列表。 |
| `rules.max_auto_retry` | 最大自动恢复次数。 |
| `rules.must_confirm_if` | 明确规定哪些情况出现时必须人工确认。 |

### 3. 系统提示词

这一类提示词通常会直接写成约束式，不给模型太多自由发挥空间。例如：

```text
你是 3C 固定工位的异常分流助手。

你的任务是根据当前异常事件、工步状态、视觉结果、执行状态、历史案例和既有规则，
输出结构化的异常判断、恢复建议和人工确认建议。

你不是主协调单元，不负责最终放行，也不负责下发底层控制指令。

请遵守以下约束：
1. 只能依据输入数据判断，不要编造不存在的现场状态。
2. 输出必须是 JSON，不要输出解释性散文。
3. 恢复建议只能从 allowed_recovery_actions 中选择。
4. 如果规则要求人工确认，不要建议自动继续执行。
5. 如果信息不足，必须明确标记 insufficient_information=true。
6. 风险等级只能使用 低、中、高。
7. 不要生成新的底层动作，不要修改 PLC、机器人轨迹或控制参数。
```

### 4. 用户提示词

用户提示词直接说明要产出什么，再把状态快照放进去就够了。例如：

```text
请基于下面的异常状态输入，输出一个结构化恢复建议包，字段必须包含：
- abnormal_stage
- risk_level
- recommended_action
- backup_actions
- requires_human_confirm
- reasons
- insufficient_information

异常状态输入如下：
{这里放上一步组装好的 JSON}
```

### 5. 最终输出最好长什么样

模型输出最好也是固定结构，不要让站控再去解析自由文本。一个更适合接到站控或 `HMI` 的结果，可以长这样：

```json
{
  "abnormal_stage": "连续吸附失败，姿态未稳定",
  "risk_level": "中",
  "recommended_action": "低速重抓",
  "backup_actions": [
    "回安全位",
    "转人工复位"
  ],
  "requires_human_confirm": false,
  "reasons": [
    "当前仍未超过最大自动恢复次数",
    "相似历史案例中，低速重抓恢复成功率更高",
    "当前工位仍处于可恢复状态"
  ],
  "insufficient_information": false
}
```

字段说明：

| 字段 | 中文说明 |
| --- | --- |
| `abnormal_stage` | 当前异常所处的更细阶段，不再只停在报警码层面。 |
| `risk_level` | 当前异常恢复的总体风险等级。 |
| `recommended_action` | 当前最推荐的恢复动作。 |
| `backup_actions` | 如果首选动作不执行或执行失败，可考虑的备选动作。 |
| `requires_human_confirm` | 是否建议转入人工确认。 |
| `reasons` | 做出当前建议的主要依据。 |
| `insufficient_information` | 当前输入是否不足以支撑可靠判断。 |

再往下走一步，规则层通常会把这份建议包收成几种固定动作，例如：

* `执行推荐动作`
* `转人工确认`
* `停止自动恢复`

### 6. 站控侧通常怎么接

站控侧最要紧的不是把模型接进来，而是先把建议包接得住。用伪代码表达，大致会像这样：

```python
def handle_abnormal_event(context):
    if context["current_snapshot"]["retry_count"] >= context["rules"]["max_auto_retry"]:
        return transfer_to_manual("retry_limit_reached")

    decision = call_agentic_ai(context)

    if decision["insufficient_information"]:
        return transfer_to_manual("insufficient_information")

    if decision["requires_human_confirm"]:
        return push_hmi_confirm(decision)

    if decision["recommended_action"] not in context["allowed_recovery_actions"]:
        return transfer_to_manual("action_not_allowed")

    return execute_recovery_action(decision["recommended_action"])
```

这段伪代码表达的是三件事：

* 先由规则层判断是否已经超过自动恢复边界
* 再让模型在受限动作集合里给建议
* 最后仍由站控决定是执行、转人工，还是停止自动恢复

这样模型和执行侧的边界就不会混掉。

## 边界与限制

这个场景里有几个边界要先说清：

* `Agentic AI` 不直接下发底层连续控制命令
* 是否执行建议，最终由主协调单元裁决
* 历史案例库沉淀的是结构化工步结果和恢复结果，不是原始大杂烩日志
* 若每次异常都依赖远程大模型，通常很难稳定满足在线预算
