# AI-first 柔性制造探索，第 7 章：具身智能里的 Agentic AI 能力与组成

摘要：这一章不再泛泛讲架构图，而是固定用一次跨站补料与空箱回收任务，把 HMI、智能体核心、技能编排、运行时、适配层、控制器与现场设备怎么接起来走一遍。重点不是把功能框列全，而是看这套分层能不能把任务理解、状态判断、恢复决策和现场执行稳稳接住，同时靠组件化和职责分离把复杂性压住。

![具身智能里的 Agentic AI 功能架构图](./diagrams/abb-isaac-agent-flexible-manufacturing-ai-first-07-agentic-ai-capabilities-and-architecture.png)

Agentic AI 不是单点模型，也不直接接管底盘、机械臂或 PLC 控制。它把任务理解、状态判断、技能编排和工业执行串起来。后面每一层都只围绕同一个任务解释，不再频繁切换例子。

先从上到下看：

* 任务和交互入口：HMI / 工单
* 智能决策中枢：智能体核心 与 技能智能编排
* 执行落地层：技能运行时 与 Robot Adapter
* 工业执行层：控制器 与 现场设备

右侧有三类横向支撑：

* 领域技能
* 状态/记忆/知识
* 感知/推理

## 问题定义

这一章的核心问题是：在“从 A 仓位取一箱连接器，送到 3 号装配站，并回收空箱”这个任务里，Agentic AI 的能力到底应该长什么样，应该放在哪一层，和其他层又是什么关系。

* 当任务通过 HMI、工单系统、API 或语音进入时，系统如何理解“现在要做什么”
* 当现场状态不断变化时，系统如何判断“现在处于什么阶段”
* 当一个动作要真正落到 ABB、ROS2、视觉系统和 PLC 上时，系统如何保证输出不是一句模糊建议，而是一个可以执行、可以约束、可以回执的动作请求
* 当系统同时面对多类设备、多种控制栈和不断增长的工艺变化时，架构如何控制复杂性、吸收异构性，并给后续扩展留出空间

从架构角度看，这里还有一组绕不过去的问题：

* 系统边界怎么切，才能把任务理解、技能编排、运行时、适配层和控制执行拆开
* 哪些能力应该做成可复用组件，避免每接一类设备、每换一个工位就重写一遍
* 组件之间靠什么组合，才能既支持当前场景，又不把后续扩展锁死
* 职责如何分离，才能让规则、状态、感知、执行和回执各自演进

## 先固定一个贯穿全文的任务

后面所有章节都只用一个固定任务往下走，把入口、判断、编排、执行和回流串成同一条链：

```text
任务：从 A 仓位取一箱连接器，送到 3 号装配站，并回收空箱
异常：去往 3 号站途中发现该站暂时占用，机器人剩余电量降到 18%
系统决定：先去临时缓存位 -> 去充电点补能 -> 重新规划路线 -> 到 3 号站交付 -> 回收空箱
```

后面无论讲哪个框，都在回答这个任务怎么穿过系统。它会经过图里最关键的几条链：

* 任务入口
* 当前状态判断
* 感知结果回流
* 技能选择与参数补全
* 运行时下发
* 控制器和现场设备执行

## 按图逐层解释

下面按分层顺序往下走，从任务入口开始，经过决策、编排和执行，最后回到状态回流。

## 1. HMI / 工单：任务从哪里进入系统

入口层负责把“给 3 号装配站补一箱连接器，并回收空箱”收成结构化任务。这一层通常由 Web HMI、API 服务和消息队列组成，负责统一接收不同来源的请求：

* 调度：上位调度系统、车队协同、站间任务分发
* 工单：MES、工单系统、生产任务开始/结束、配方切换
* API：其他业务系统直接下发请求
* Voice：人工口头指令、班组长或运维确认

无论来自哪里，入口层最终都输出统一任务对象，至少包括：

* 任务目标
* 起点 / 终点
* 回收或附加要求
* 优先级与时限
* 来源身份
* 当前约束条件（如最低安全电量、站点状态等）

本例形成的结构化任务可能是：

* 给 3 号装配站补一箱连接器
* 从 A 仓位取料，送至 3 号装配站
* 同时回收空料箱
* 任务需在规定时限内完成
* 电量不得低于安全阈值后再执行任务

## 2. 领域技能：哪些能力来自行业和工艺

领域层不做通用推理，只负责提前定义“跨站补料并回收空箱”这类任务的行业规则、站点规则和可执行技能。通常以 YAML / JSON 配置、数据库和技能注册表实现。

其中，领域定义回答：

* 服务什么业务
* 任务对象是什么
* 成功 / 失败如何判定
* 工艺、站点和角色边界是什么

领域技能回答：

* 系统允许做哪些动作
* 针对不同料号、工艺、设备可以调用哪些策略

对于“从 A 仓位补到 3 号站并回收空箱”这个任务，领域层需要提前固定：

* 地图语义：A 仓位、3 号站、缓存位、充电点、禁行区分别代表什么
* 通行规则：哪些路线可走、哪些区域限速或限时
* 任务模板：取料 → 送达 → 回收空箱
* 电量策略：低于多少必须先充电
* 站点规则：什么状态允许交付，空箱何时算回收完成

因此，后续编排层只能在预定义能力内执行，例如：

* 去取料位取货
* 去缓存位等待
* 去充电点补能
* 重新规划路线
* 到站交付并回收空箱
* 转人工确认

## 3. 智能体核心：系统在这里形成“理解”和“判断”

这一层是 Agentic AI 的决策核心。它负责理解“这是不是一次完整的跨站补料闭环”、判断当前卡在哪个阶段，并决定下一步该走缓存、补能、重规划还是转人工，但不直接执行。

首版实现更适合用 大语言模型 + 领域技能。LLM 负责理解任务、整理现场上下文、生成候选决策，领域技能 负责把可执行能力、站点边界和恢复路径收成受限集合。

智能体核心真正起作用的是：

* 任务理解：把入口层请求翻成系统内部可操作的任务目标。
* 长程规划：在任务级别拆分阶段、选择路径、安排恢复链路。
* 状态判断：结合感知结果、设备状态、工艺约束和历史上下文，判断当前处于什么阶段。
* 后续决策：给出下一步建议，例如优先调用哪类技能、补哪些参数、是否继续自动恢复。

### 3.1 任务理解

任务理解要把入口层送来的补料请求翻成系统内部可操作的目标，而不是只把它当成一条普通配送命令。先要弄清：

* 用户说的是什么任务
* 任务属于哪个模板
* 当前任务和历史上下文怎么关联
* 任务完成判据是什么

在本例里，它会把请求识别成“一次跨站补料并回收空箱的任务闭环”，而不是普通导航或单纯取货动作。

### 3.2 长程规划

长程规划说的是补料闭环的任务级规划，不是底层轨迹规划。它关心的是：

* 这件事大致应该分成哪些阶段
* 当前阶段之后有哪些候选路径
* 遇到异常时应往哪条高层恢复链路走

例如：

* 是继续当前工步
* 是切到恢复流程
* 是先去缓存位等待后重进任务
* 是转人工确认

在本例里，长程规划关心的不是末端执行器怎么走轨迹，而是当前该走哪条恢复链：

* 直接继续去 3 号站
* 先去临时缓存位等待
* 先去充电点补能后再来
* 转人工确认

### 3.3 状态判断

状态判断最关键，因为系统要区分“只是 3 号站暂时占用”还是“占用、电量和路线风险已经叠加到不能继续自动执行”。它要回答：

* 现在系统到底处于什么状态
* 当前异常属于哪个阶段
* 当前设备和物料条件是否允许继续
* 当前风险是工艺风险、设备风险，还是节拍风险

在本例里，状态判断要给出类似这样的结论：

* 当前阶段是“跨站运输中的异常恢复阶段”
* 3 号站暂时不可交付
* 当前电量 18%，低于安全送达阈值
* 当前更像资源占用和续航约束叠加，不像底盘故障

### 3.4 后续决策

后续决策给出下一步建议，但不是自由生成底层动作。它输出的更像是：

* 下一步优先推荐的技能类别
* 需要补充什么参数
* 是否要请求新的感知结果
* 是否应直接触发人工确认

边界很明确：

* 它可以决定方向
* 但它不应直接越过下层去控制设备

在本例里，后续决策给出的不应是底层速度曲线，而应是：

* 优先建议先去临时缓存位
* 如果电量不足以安全等待和继续交付，则插入充电动作
* 然后重新规划路线，再去 3 号站完成交付和空箱回收

## 4. 状态/记忆/知识：系统为什么不会只看当前一帧

这一层是状态和知识底座，负责把“车现在在哪、3 号站是不是被占、18% 电量够不够、以前这种情况怎么收尾”稳定送回决策链。落地上通常是事务型数据库 + 日志存储 + 对象存储的组合。

主要包括：

* 状态快照：记录当前工步、阶段、异常码、站点占用结果、机器人模式和重试次数。
* 历史案例：沉淀带结果的相似补料案例，帮助系统判断哪条恢复路径成功率更高。
* 日志回放：保留关键事件、输入、决策和回执，方便还原现场。
* 工艺配方：保存等待时长、最低电量阈值、路线限制和允许动作范围。
* 产品配置：描述当前运送的连接器类型、目标站点配置和现场设备差异。

### 4.1 状态快照

状态快照记录的是当前时刻的结构化系统状态，例如：

* 当前工步：说明任务此刻正落在哪一个具体工步上，例如取货、出库、跨站运输、到站交付或空箱回收。
* 当前任务阶段：说明系统正处在主流程、恢复流程、等待回执还是人工确认阶段，帮助上层判断接下来该走哪条路径。
* 当前异常码：用统一编码记录当前故障或异常类型，避免不同设备各说各话，方便后续判断和统计。
* 当前视觉结果：记录这一轮视觉系统给出的核心输出，例如目标站是否占用、对接标记是否可见、缓存位是否空闲、识别结果是否通过。
* 当前机器人模式：说明机器人当前处于自动、手动、暂停、故障、恢复或安全模式，决定系统是否还能继续自动执行。
* 当前重试次数：记录当前动作或当前阶段已经自动重试了几次，用来判断是否还允许继续恢复，还是该转人工或终止任务。

### 4.2 历史案例

历史案例不是普通日志，而是带结果、能比较的经验沉淀。它主要回答：

* 当前问题能不能匹配到相似历史：判断这次异常和过去哪些案例足够接近，能不能直接借用已有经验，而不是从零开始决策。
* 哪条恢复路径曾经成功：比较不同恢复动作在历史上的结果，看看等待、充电、重规划或转人工哪种更可靠。
* 哪类连接器、哪类站点组合更容易在什么阶段失败：沉淀出“某些物料、某些目标站、某些运输阶段更容易出问题”的模式，帮助系统提前提高警惕。

### 4.3 日志回放

日志回放让团队能回答：

* 当时系统看到了什么：把当时的输入、状态、视觉结果和设备回执还原出来，避免事后只能靠回忆推测。
* 为什么选了这条路径：让团队看到当时的判断依据和决策过程，知道系统为什么没有走别的分支。
* 这次失败到底是感知错、判断错，还是执行错：把问题拆回具体环节，判断根因是在识别、决策还是底层动作执行上。

### 4.4 工艺配方

工艺配方决定的是参数边界和工艺资格，例如：

* 当前连接器料号允许的补料规则：说明这类物料对应的最低库存、交付时限或安全送达阈值，超出范围就不能继续执行。
* 从 A 仓位到 3 号站这条链的等待和补能条件：规定允许等待多久、需要满足什么电量或站点状态后才能进入下一步。
* 当前恢复动作是否允许自动执行：明确先去缓存位、先去补能还是再次尝试交付哪些可以自动做，哪些在当前任务里根本不允许，必须转人工。

### 4.5 产品配置

产品配置决定的是产品和任务模板的差异，例如：

* 当前运送的连接器版本：说明这次补的是哪一类连接器，不同版本可能对应不同交付优先级、包装规格和站点要求。
* 料箱与空箱的组合关系：描述这次任务要求送去什么料箱、回收什么空箱，避免系统送对了料却回错了箱。
* 3 号站是否启用特定交付能力：说明当前目标站有没有打开自动对接、到站确认或空箱回收联动能力。
* 当前部署现场有哪些可用设备：记录现场到底接了哪些移动机器人、相机、充电桩和站点 PLC，系统才能知道当前能调用哪些能力、不能依赖哪些能力。

合起来看，这一列提供的是稳定上下文。对本文这个任务，系统至少要知道：当前在从 A 仓位去 3 号站的运输段，3 号站暂时占用，当前电量 18%，恢复次数为 0，当前产品配置和工艺配方分别对应哪类连接器、哪类站点以及最低安全送达电量、允许等待时长、是否允许先去缓存位。历史案例和日志回放再补一句更实际的话：以前遇到“目标站占用 + 电量逼近阈值”时，走“先缓存 -> 充电 -> 重新规划 -> 再交付”是否更稳。

## 5. 技能智能编排：把“想做什么”收成“准备用什么技能做”

技能编排层把“先缓存、再补能、再重规划、再去 3 号站”这种高层建议收成受限技能、明确参数和可执行流程。

这一层最适合用 技能注册表 + 规则引擎 + 行为树/状态机 落地。对本文这个移动机器人场景来说，行为树比通用工作流更贴近执行编排，尤其适合条件分支、回退和恢复。

图中央第三层是 技能智能编排，主要收敛发生在这里：

* 技能选择
* 参数补全
* 行为规划
* 约束检查

它位于智能体核心和技能运行时之间，把建议收成受限技能和可执行流程。

### 5.0 技能到底长什么样

前面一直在提“技能”，这里先把它写实一点。技能不是一句自然语言命令，也不是一段临时生成的动作文本。一个能编排、能执行、能回执的技能，至少要有这些字段：

* `skill_id`
* 输入参数
* 前置条件
* 成功条件
* 失败条件
* 可调用的 `Adapter`
* 可选恢复路径

技能对象通常会长成这样：

```json
{
  "skill_id": "deliver_material_to_station",
  "requires": [
    "material_loaded",
    "station_03_known",
    "battery_check_passed"
  ],
  "inputs": {
    "pickup_location": "rack_a_12",
    "target_station": "station_03",
    "buffer_location": "buffer_b_02"
  },
  "success": "material_delivered && empty_box_loaded",
  "failure": [
    "station_occupied",
    "battery_below_threshold",
    "route_blocked",
    "timeout"
  ],
  "adapters": [
    "ROS2",
    "Vision",
    "PLC"
  ],
  "recover": [
    "buffer_then_retry",
    "charge_then_retry",
    "manual_confirm"
  ]
}
```

放到本文这个跨站补料例子里，这个定义已经把几件关键事定住了：调用的是“去目标站交付并回收空箱”这类稳定技能，进入执行前必须满足物料已装载、目标站点已知、电量检查通过，运行时则根据成功条件、失败条件和恢复路径判断下一步该继续、重试还是转人工。

技能智能编排 和 技能运行时 的关系也很清楚：

* 编排层决定“选哪个技能、补哪些参数、走哪条恢复路径”
* 运行时负责“把这个技能对象映射到真实控制接口，再把结果拿回来”

### 5.1 技能选择

技能选择要把上层给出的后续决策映射成具体技能类别，而不是停留在“换条路试试”这种模糊建议上。

例如，它要判断：

* 当前更适合先去缓存位还是先去充电点
* 当前应走“继续交付”还是“转人工”
* 当前这一步应选导航、缓存、补能、交付还是空箱回收技能

### 5.2 参数补全

参数补全要把补料闭环每个技能执行所需的参数拼齐，例如：

* 目标工位
* 取料位、缓存位和目标站点
* 路径约束和等待时长
* 最低安全电量阈值
* 当前配方版本

没有这一步，技能只是个名字，无法执行。

### 5.3 行为规划

行为规划可以直接理解成行为树编排。走到这里，系统不再只是决定“用哪个技能”，而是把这次补料任务里的技能、条件检查、恢复动作和 fallback 组织成一棵能执行的树。

在机器人场景里，行为树适合表达这些结构：

* 顺序执行
* 条件判断
* 重试与回退
* 感知失败后的重新观察
* 主路径失败后的恢复分支

放到本文这个跨站补料例子里，一棵典型行为树大致是这样：

* 先检查物料是否已装载、目标站是否可服务
* 判断当前电量是否足够直接完成任务
* 如果目标站暂时占用，先去临时缓存位等待
* 如果等待后电量逼近阈值，则插入充电动作
* 然后重新规划路线，再去 3 号站交付并回收空箱
* 如果仍不满足条件，就转到人工确认

到这里，高层建议会被收成一棵可执行的行为树，而不是线性动作列表。运行时执行的是树里的叶子节点，并根据 `success / failure / running` 持续 tick 整棵树。

如果想把“为什么这里更适合用行为树，而不是流程图或状态机”单独展开，可以继续看：[第 7 章补篇：为什么行为规划更适合用行为树](./abb-isaac-agent-flexible-manufacturing-ai-first-07-behavior-planning-with-behavior-tree.md)。

### 5.4 约束检查

这个约束检查更偏决策前检查，用来判断：

* 当前技能是否被允许
* 当前参数是否在工艺范围内
* 当前阶段是否允许自动执行
* 是否必须先走人工确认

这一层还没进到底层运行时，但允许的动作空间已经收紧了。本例里，抽象建议会在这里收成一个受限方案：

* 技能选择 选中“先去缓存位等待，再决定是否充电和重规划”
* 参数补全 补齐缓存位编号、充电点编号、最低安全电量、目标站等待上限
* 行为规划 形成 `缓存 -> 电量复核 -> 充电 -> 重新规划 -> 到站交付 -> 回收空箱`
* 约束检查 判断当前是否允许自动等待、自动充电和再次尝试交付

## 6. 感知/推理：系统如何理解物理世界

感知与推理层通过视觉模型、空间智能和世界模型，把“站点是否可交付、缓存位是否可停、路线和充电是否可行”等现场观测转换成决策可用的物理状态。

### 6.1 视觉模型

视觉模型 负责把图像和传感器信息变成结构化结果。

例如：

* 目标站是否可识别
* 对接标记是否可用
* 缓存位或交付位是否空闲
* 当前环境里有哪些障碍、占用或异常状态

### 6.2 空间智能

空间智能在视觉结果之上进一步回答：

* 还能不能走
* 能不能停靠
* 有没有可达性问题
* 当前绕行或等待是否还在可接受范围

### 6.3 世界模型

世界模型关注的是状态如何演化，回答的是：

* 如果执行这个动作，下一步大概率会怎样
* 当前状态会不会继续恶化
* 某条恢复路径的后果是什么

放到这个例子里：

* 视觉模型识别到 3 号站当前被占用、缓存位是否空闲、对接标记是否可见
* 空间智能判断当前路线是否可达、缓存位是否可停、充电点是否能安全插入当前任务
* 世界模型评估“继续直奔 3 号站”失败概率较高，而“先缓存 -> 充电 -> 再规划”更稳

## 7. 技能运行时：把编排结果落到系统接口

这是整张图里最偏工程实现的一层。技能运行时不再做高层判断，它做的是把“去缓存位 -> 去充电点 -> 重规划 -> 到站交付 -> 回收空箱”稳定落到底层接口。工程上通常做成独立运行时服务，输入是技能对象或动作对象，输出是结构化回执。

### 7.1 Robot Adapter 和控制器之间到底传什么

这一层传的不是自然语言，也不是“给 3 号站送料”这种高层任务描述，而是结构化动作对象。

边界很简单：

* Robot Adapter 不接收自然语言
* 它也不直接接收高层任务描述
* 它接收的是结构化动作对象，也可以叫动作协议对象

这个对象要把“这一次怎么执行”说清楚，至少要包含：

* `action_id`
* `action_type`
* 目标对象或目标位姿
* 约束参数
* 前置条件
* 成功判据
* 超时和失败策略
* 目标 `Adapter`

对本文这个补料恢复流程，运行时发给 Robot Adapter 的对象可以长成这样：

```json
{
  "action_id": "act_deliver_material_retry_02",
  "action_type": "buffer_charge_reroute_delivery",
  "target": {
    "pickup_location": "rack_a_12",
    "buffer_location": "buffer_b_02",
    "target_station": "station_03",
    "charge_dock": "charge_01"
  },
  "constraints": {
    "min_battery_arrival_pct": 25,
    "max_wait_station_ready_s": 120,
    "route_policy": "avoid_blocked_corridor"
  },
  "preconditions": [
    "material_loaded",
    "buffer_available",
    "plc_interlock_released"
  ],
  "success": {
    "material_delivered": true,
    "empty_box_loaded": true
  },
  "timeout_ms": 180000,
  "on_failure": [
    "charge_then_retry",
    "wait_then_retry",
    "manual_confirm"
  ],
  "adapter": "ROS2"
}
```

这里和前面的技能对象要分开看：

* 技能对象描述“系统允许做什么”
* 动作对象描述“这一次具体怎么做”

编排层输出的是技能，运行时真正下发的是动作协议对象。先把这一层标准化，后面的 ROS2、Vision、充电接口和 PLC 才能接在同一条执行链上。

### 7.2 技能映射

技能映射负责把上层的技能名映射成具体系统中的动作实现。

例如：

* “去缓存位等待”映射到哪类导航任务
* “去充电点补能”映射到哪个充电接口
* “到 3 号站交付并回收空箱”映射到哪组停靠、握手和回收动作

### 7.3 参数绑定

参数绑定把上层给出的参数绑定到具体控制接口的字段格式。这里做的是接口级翻译，例如：

* 上层的目标站点、缓存位和充电点转成控制器参数
* 上层的等待上限和最低安全电量转成运行时约束
* 上层的任务标识转成 PLC 或站控可识别字段

### 7.4 约束检查

这里的约束检查和上层编排阶段不一样，更偏执行前检查：

* 控制器是否在线
* 当前设备状态是否允许下发
* 参数是否满足接口和安全前置条件
* 当前互锁是否解除

* 编排层的约束检查决定“逻辑上该不该做”
* 运行时的约束检查决定“现在能不能发出去”

### 7.5 执行调度

执行调度负责组织“先去缓存位还是先去补能、什么时候等回执、什么时候继续重规划”的调用顺序、超时、并发和回执。它要保证：

* 哪些命令先发
* 哪些命令等待结果再继续
* 超时后如何处理
* 失败后如何上抛

### 7.6 Robot Adapter

Robot Adapter 是这套架构的接口隔离层。系统不直接耦合到底层硬件，而是通过适配层对接不同接口。这样设计有几个直接结果：

* 上层能力可以尽量保持统一
* 不同现场的设备差异留给适配层处理
* 新设备、新接口、新控制栈的接入成本更可控

放到这个例子里，落到系统接口时，会被翻译成这样的请求：

* 技能映射 把“先去缓存位等待”映射成一次导航任务，把“去充电点补能”映射成一次充电任务
* 参数绑定 把缓存位编号、目标站点、等待时长和最低安全电量写进控制字段
* 约束检查 确认移动底盘在线、站点互锁状态可读、PLC 互锁已解除
* 执行调度 按顺序触发 `ROS2 导航 -> Vision / 定位确认 -> 充电 -> ROS2 重新规划 -> 到站交付`

接着，Robot Adapter 会把这些调用分别转成：

* 给 Vision 的站点占用确认或对接标记识别请求
* 给 ROS2 或移动底盘控制栈的导航、停靠、充电动作
* 给 PLC 的互锁确认、到站握手或工步切换信号

它实际做的是：

* Robot Adapter 先接收结构化动作对象
* `ROS2 Adapter` 再把它翻译成导航、停靠、充电任务或厂商接口请求
* `PLC Adapter` 再把它翻译成 `IO`、`OPC UA`、`Modbus` 或站控字段更新
* `Vision Adapter` 再把它翻译成具体视觉触发、重采集或检测调用

只有把这层定义清楚，`技能运行时 -> Robot Adapter -> 控制器` 才是一条能实现的软件接口链。

## 8. 控制器与现场设备：真正执行发生在哪里

到了执行边界，控制器运行已验证的控制逻辑，现场设备把“去 A 仓位取料、去缓存位等待、去充电点补能、到 3 号站交付、回收空箱”这些动作做出来。落地原则也很简单：控制器沿用现有工业控制栈，现场设备继续由自动化系统接管，上层不要越过这条边界。

### 8.1 控制器

控制器这一侧对应的是：

* AMR 控制器
* ROS2
* 站点 PLC

这里承载的是设备控制逻辑本身，负责：

* 执行已验证控制程序
* 管理底层动作逻辑
* 处理底层回执和状态

### 8.2 现场设备

现场设备这一侧对应的是：

* 移动底盘
* 3 号站对接机构、临时缓存位和充电桩
* 站点传感器与 PLC I/O

再往下就是物理世界本身。动作有没有真的发生、传感器有没有真的返回正确状态，最后都在这里见分晓。

放到这个例子里，现场真正发生的事情是：

* AMR 控制器、ROS2 或底盘控制器执行导航、停靠和充电动作
* 视觉系统回传站点占用、对接标记和缓存位状态
* 移动机器人完成到站交付，并装回空箱
* PLC 返回到站交付完成或失败状态

直到这一步，任务才从“系统建议”变成“现场结果”。

## 9. 最值得先验证什么

首版实现不该平均铺开，先验证最容易决定这条补料闭环能不能成立的几个接口和闭环：

1. HMI / 工单 的输入能不能稳定收成“从 A 仓位到 3 号站并回收空箱”的结构化任务包，而且这个任务包能不能被后续模块复用。
2. 状态快照、工艺配方、产品配置 能不能稳定表达“3 号站占用 + 电量 18% + 当前在运输阶段”这类关键上下文，并和决策层、编排层解耦。
3. 技能选择、参数补全、行为规划 能不能稳定收成 `缓存 -> 补能 -> 重规划 -> 交付 -> 回收` 这条固定技能链，让技能真正可组合、可复用。
4. 技能运行时 到 Robot Adapter 的接口能不能稳定映射到 ROS2、Vision、PLC 和充电接口，并支持后续扩展新站点而不改上层逻辑。
5. 底层回执能不能把“站点是否释放、补能是否完成、交付是否成功、空箱是否装回”回流成 历史案例 与 日志回放，同时保持统一回执格式，别让不同设备把状态层打散。

其实就是在验证三件事：

* 组件边界是不是清楚
* 组件之间的契约是不是稳定
* 新场景接进来时，系统到底是在“扩展组件”，还是又回到“改一个大模块”
