## 摘要

这是一篇研发视角的复盘。

最近我们根据需求人员给出的 PRD 做规格化开发，再把规格拆成研发任务交给 AI Coding。这个过程更像一次共同摸索：过去可以在评审、开发和测试中逐步磨合的边界，现在需要更早被写清楚，比如范围怎么收、旧规则怎么处理、跨模块影响怎么落到可验收的行为里。

过去，PRD 主要写给人看。研发读完，开会确认，再慢慢拆实现。现在这条链路更短：需求人员写 PRD，PRD 规格化，规格拆成研发任务，AI 按研发任务写代码和测试。中间如果有模糊点，AI 不会停下来等团队慢慢磨，它会先写出一版实现。

这篇文章复盘两次规格化开发。

第一组是“物理环境管理”。需求侧给了 PRD，也给了两份 UI HTML 原型。我们在规格化时补了一份工程边界文档，把目标、非目标、数据归属、状态规则和验收场景整理得更适合进入开发。

第二组是“设备实例与设备模板管理”。规格化之后，一个需求 PRD 被拆成两个研发边界：资源中心里的设备实例和设备模板管理，以及智能体如何通过物理环境获得设备能力。

这两次经历给我的感受很直接：如果目标是让 AI 继续拆研发任务并写代码，需求 PRD 不能只写“要做什么页面、有什么功能”。它还要写清楚哪些不做，旧规则怎么处理，跨模块关系怎么变，哪些行为能验收，哪些内容会影响研发任务拆分。

**本文想讨论的是：研发在根据需求 PRD 做规格化开发时，遇到了哪些问题，以及如何反向改进 PRD 规范，让需求 PRD 本身逐步达到可拆分研发任务的质量。**

## 引言：规格化时最先碰到的不是代码问题

研发拿到 PRD 后，第一步通常不是写代码，而是把需求变成可执行规格。

这一步会做几件事：把目标和非目标拆开，把业务规则整理成状态和行为，把跨模块影响找出来，把验收口径改成能测试的场景。做完这些，后面才好拆研发任务，交给 AI Coding。

问题也集中暴露在这里。

需求不清楚，过去也会造成摩擦。

评审时没讲透，研发阶段再问。实现到一半发现状态规则缺一块，再拉人确认。测试发现边界没定义，再回头补口径。这个过程慢，但团队还能靠项目节奏消化。

AI Coding 之后，这种消化空间小了很多。

沟通没有变复杂，执行变快了。过去两天后才暴露的问题，现在十几分钟就可能变成一版代码。团队再围着这版代码讨论：这里要不要级联解绑，那里能不能重复选设备，某个“后续规划”到底是不是本期范围，旧数据要不要兼容。

看起来是 AI 写快了。实际上是模糊成本被提前结算了。

人读 PRD，会带着经验补空白。AI 没有团队默契。它会把显性规则、历史旧描述、未来扩展和隐含冲突一起吃进去，然后很快写出结果。

所以，研发做规格化时要解决的重点，不只是把 PRD 改写成技术文档，而是判断这份 PRD 能不能进入“拆研发任务 -> AI 开发 -> 行为验收”的执行链路。

## 一、研发在规格化什么

从研发视角看，规格化处理的是需求到执行之间的转换：

1. 需求人员输出可拆分研发任务的 PRD 和 UI 原型。
2. PRD 通过规格自检，确认边界足以进入研发任务拆分。
3. AI 或研发任务拆分工具根据 PRD 生成研发任务。
4. AI 按研发任务实现代码和测试。
5. 团队用 PRD 的行为验收和 UI 原型的体验要求验收结果。

PRD 在这条链路里要多承担一层职责。它不只是需求说明，也是后续研发任务拆分的上游输入。

它仍然要表达用户目标、业务语境、页面结构、交互路径和体验预期。同时，它还要把目标、非目标、业务规则、边界情况、数据状态、验收标准和影响域写到能拆研发任务的程度。

这不要求需求人员写技术方案。代码怎么组织、架构怎么取舍、研发任务质量怎么校验，仍然是研发的职责。研发希望前移的是需求决策，不是实现方案。需要避免的是另一种卡点：每个需求都先开一轮同步边界评审，再进入拆分。

更合适的做法是把边界问题沉淀进 PRD 模板和规格自检里。研发负责制定标准、抽查结果、校验研发任务拆分质量，并在边界冲突时兜底。

## 二、规格化阶段不能只靠人工兜底

过去很多模糊点可以靠会议、口头确认和研发经验兜住。进入 AI Coding 后，这些方法还在，但效果会变弱。

| 旧方式 | 在 AI Coding 中的问题 |
|---|---|
| 靠需求评审兜底 | 评审能发现一部分问题，但未固化的口径很快会变成代码 |
| 靠研发阶段追问 | 每个研发任务都回头问需求，AI 提速会被沟通返工吃掉 |
| 靠长 PRD 传递上下文 | 长文档保留了语境，但 AI 很难判断硬规则、体验描述和未来规划的优先级 |
| 靠 AI HTML 原型表达体验 | 初版生成快，后续局部修改难稳定 |
| 靠人工验收发现问题 | 验收能发现结果错了，但跨模块契约和旧数据清理发现得越晚，返工越重 |
| 靠研发自己拆研发任务 | 人能凭经验拆边界，AI 更容易按页面、按钮、接口这些显眼对象切分 |

### 2.1 前置规格自检

需求评审过去常常承担“兜底澄清”的职责。AI Coding 里，如果每个需求都必须等需求和研发同步评审后才能拆研发任务，评审本身就会卡住节奏。

把评审问题前置成规格自检项更稳：状态机是否完整，删除联动是否清楚，数据归属是否明确，旧规则是否废弃，发布兼容是否有口径，异常路径是否能验收。

这样 PRD 完成时，已经先回答了一轮“能不能拆研发任务”的问题。研发可以抽查规格质量和研发任务拆分结果，而不是默认参与每个需求的前置澄清。

### 2.2 区分信息权重

传统 PRD 往往很完整，背景、术语、页面、文案、弹窗、操作反馈、验收标准都有。这些内容对人有用。

AI Coding 还需要优先级。

同一份 PRD 里会同时出现两类信息：

| 类型 | 示例 | 对研发任务拆分的影响 |
|---|---|---|
| 系统规则 | 只有已激活环境可被智能体选择；已激活环境任一角色设备被删除后立即转草稿；同一环境内重复选择同一设备实例只做非阻断提示 | 决定状态机、数据校验、异常路径和测试用例 |
| 体验表达 | 卡片怎么展示；按钮怎么排列；Toast 文案怎么写 | 决定页面呈现、交互反馈和视觉一致性 |

两者都重要，但进入执行链路时不能混在一个权重里。

这对前后端分离团队尤其明显。系统规则通常会落到后端状态机、数据模型、权限校验、接口契约和测试用例里；体验表达更多落到前端组件、交互状态、文案反馈和视觉一致性里。如果 PRD 没有把这两类信息分开，研发任务拆分时就容易出现三种情况：前端任务带着后端规则一起做，后端任务漏掉页面状态反馈，或者同一条规则在前后端各理解一遍。

规格化时要做的事情，就是把这些内容拆清楚：哪些是跨前后端都必须遵守的产品规则，哪些是前端体验要求，哪些是后端契约，哪些需要单独变成联调和验收任务。

### 2.3 管理原型变更

这次需求侧给了 UI HTML 原型。它的价值很明显：需求人员能很快把想法变成可见界面，团队也能围着一个可运行页面讨论。

难点出现在后续修改。需求人员可以让 AI 很快生成一个初版，但要让 AI 按自己的想法稳定改动局部 UI，就没那么容易。

需求人员通常不是前端开发，也不是 UI 设计师。他们可以说“这个弹窗太重”“筛选区太挤”“卡片层级不清楚”，但很难说清楚结构化 UI diff：改哪个组件，保留哪些布局，不要动哪些区域，交互逻辑是否保持，假数据是否同步，样式变化会不会影响其他页面。

AI 接到模糊修改请求后，容易把局部调整做成整体重构。它可能顺手改结构、文案、交互、样式变量，甚至改掉和 PRD 规则相关的状态展示。

所以，AI 生成的 UI 原型更像可运行草稿，不是稳定设计稿。它降低了早期表达门槛，也把问题从“画不出来”变成了“改不准确”。

UI 原型也需要变更边界。每次改动要说清楚：改哪里，保留哪里，哪些交互不变，哪些状态要验收。不然原型多改几轮后，会和 PRD、研发任务一起漂。

### 2.4 写清变更差异

还有一类问题在规格化时很常见：PRD 讲清了新功能长什么样，但没有说清它相对旧系统改了什么。

新建模块这样写通常够用。变更类需求就不够了。它除了描述新增页面、新字段、新流程，还要说清哪些旧能力被删除，哪些旧入口被替换，哪些旧规则废弃，哪些旧数据要迁移，哪些旧接口继续兼容，哪些关系模型从此不再成立。

这些变化如果没有写出来，AI 只能猜。它可能保留一个本该删除的入口，也可能继续实现一条已经废弃的旧规则，还可能把“新增能力”理解成“在旧模型上叠一层页面”。对研发来说，这些都不是代码细节，而是规格化阶段必须先定下来的变化边界。

## 三、两次规格化暴露了什么

### 3.1 物理环境案例

第一组是“物理环境管理”。需求侧材料包括一份 PRD 和两份 UI 原型：

- `PRD.md`
- `UI--资源中心--物理环境.html`
- `UI--智能体--配置物理世界.html`

这组材料不是“只有文字，没有体验”。PRD 已经写了业务背景、术语、模块关系、状态机、环境类型过滤、设备角色映射、环境与智能体关联、设备删除级联影响、页面入口、页面结构、全局交互、信息实体和验收标准。

UI 原型也覆盖了很多细节。资源中心原型包含物理环境列表、筛选、状态汇总、环境卡片、详情页、创建页、编辑页、删除确认、激活校验弹窗、Toast、重复设备提示、关联智能体浮窗。智能体配置原型覆盖智能体列表、物理环境 Tab、仅展示已激活环境、环境勾选，以及左侧智能体卡片同步显示。

所以，体验层并不空。

规格化时，我们主要补的是工程边界：把尚未固定的内容整理成目标、非目标、规则、数据归属、实现约束、测试口径和 Given / When / Then 验收。这份补充文档像一份反向样本，告诉我们需求 PRD 里还缺哪些东西。

### 3.2 设备管理案例

第二组是“设备实例与设备模板管理”。这里更值得注意：规格化之后，一个需求 PRD 被识别出两个工程边界。

- `设备实例与设备模板管理功能 PRD`：处理资源中心设备管理本身，包括设备类别、设备类型、在线/离线状态、设备模板 Schema、删除影响预览、设备类型字典引用、设备实例删除级联。
- `智能体通过物理环境获得设备能力 PRD`：处理智能体能力来源模型，清理智能体直接绑定设备实例的旧链路，统一为 `智能体 -> 物理环境 -> 环境角色 -> 设备实例`，并约束运行时设备上下文。

### 3.3 工程边界拆分

需求人员通常按产品入口组织材料。研发做规格化和任务拆分时，看的是另一套边界：工程所有权、数据模型、运行时链路、旧数据清理、跨模块契约和验收边界。

一个需求 PRD，不一定只对应一个研发任务包。

## 四、规格化时反复补的是六类边界

两组材料放在一起看，问题不在信息量。很多内容其实已经写了。规格化时反复补的，是那些会影响工程执行的边界决定。

### 4.1 非目标边界

需求 PRD 会写“不包含”一些大能力，比如克隆、导入导出、运行态监控、完整 Real2Sim2Real 闭环。规格化时，我们又进一步明确：不做环境模板、版本管理、批量操作、设备实例深度编辑、设备实例批量删除、跨环境冲突判断、旧直接绑定长期兼容。

这些非目标应该前置。AI 很容易把“后续扩展”“看起来顺手”“系统里可能需要”的内容提前做掉。

### 4.2 旧规则与变更差异

规格化文档明确覆盖了一些旧口径。例如设备实例删除不再按“最后一个有效设备实例删除才转草稿”，而是任一已激活环境角色引用的设备实例被删除即转草稿；设备类型不保留实例侧快照，而是通过模板实时读取；旧智能体直接设备绑定模型要清理，不再作为正式关系。

AI 不会自动判断哪段是旧描述，哪段是新结论。旧规则还在上下文里，就可能被实现进去。

这也是变更需求最容易漏的部分。新建模块的核心问题是“要建什么”，变更需求的核心问题是“旧系统怎么变”。如果 PRD 只描述升级后的新界面，研发就必须回头追问：旧字段还要不要兼容，旧数据怎么处理，旧 API 是否保留，旧状态如何映射，旧绑定关系是否阻断删除。

| 需求类型 | PRD 重点回答 | 容易遗漏的内容 |
|---|---|---|
| 新建模块 | 入口在哪里、有哪些对象、有哪些流程、怎样验收 | 初始状态规则、边界情况、非目标 |
| 变更需求 | 旧能力如何保留、修改、删除、替换、迁移、兼容 | 被删除的入口、废弃规则、旧数据处理、旧接口影响、用户从旧流程进入新流程时看到的变化 |

“设备实例与设备模板管理”就是变更需求。它并非单纯新增几个字段，而是在纠正旧概念：设备模板上的“设备类别”要改成“设备类型”，设备实例状态不能继续混用激活、在线、执行中、异常，智能体直接绑定设备实例的旧模型要被物理环境模型替代。

### 4.3 跨模块契约

物理环境连接设备管理模块和智能体设计模块。设备实例删除会影响环境状态，环境状态变化会影响智能体关联，智能体运行时设备上下文又依赖物理环境角色。

这些不是研发实现细节，是产品行为。用户删除设备时系统如何解释影响，智能体失去设备能力时是否自动解绑，旧直接绑定数据是否还阻断删除，都会改变用户预期。

### 4.4 数据归属

设备实例由设备管理维护，物理环境只读引用设备实例，智能体通过物理环境间接获得设备能力。设备类型来自模板或字典，设备状态由运行态驱动，前端不提供手动编辑入口。

这些决定了用户在哪里编辑数据、在哪里看到关联、哪里不能跳转、哪里不会合并能力。需求不写清楚，研发只能在实现前反复确认。

### 4.5 行为验收

“功能正常”“信息正确”“交互流畅”对人工验收有点用，对 AI Coding 不够。规格化文档里的 Given / When / Then 更容易进入研发任务拆解和自动化测试：草稿允许设备为空，已激活不允许设备为空；真实环境只展示物理设备；删除被已激活环境引用的设备后，环境转草稿且智能体解绑。

### 4.6 发布和异常处置

AI 可以很快写功能，但上线后的兼容说明、旧数据清理、异常降级和回滚策略仍然要提前定。智能体从直接绑定设备实例切换为绑定物理环境，改的是关系模型，不只是页面。

## 五、可拆分 PRD 是研发任务的上游输入

### 5.1 结构影响拆分

PRD 不是最终交付物。它通常会继续进入研发任务拆解。

研发会根据 PRD 让 AI 生成拆解后的开发内容。PRD 的结构，会直接影响 AI 拆出来的研发任务。

如果 PRD 只描述页面和新增功能，AI 很容易按显性对象拆：列表页一个研发任务，详情页一个研发任务，创建编辑一个研发任务，智能体 Tab 一个研发任务。

这种拆法会漏掉需要单独承接的系统行为：

- 状态机和状态转换。
- 删除影响预览。
- 设备实例删除级联。
- 物理环境状态降级后的智能体解绑。
- 旧直接设备绑定数据清理。
- 运行时设备上下文派生。
- 字典引用保护。
- 发布兼容与回滚口径。

### 5.2 骨架决定质量

PRD 要能支撑研发任务拆分，就要给 AI 一个稳定骨架。

`目标 / 非目标` 决定哪些研发任务应该出现，哪些不该出现。`业务规则` 决定哪些研发任务共享同一套状态约束。`边界情况与错误状态` 决定异常路径会不会漏。`数据与状态` 决定数据归属和模块边界。`实现决策` 限制 AI 不能自行扩展或改写的系统结构。`测试决策` 告诉每个研发任务完成后如何证明它做对了。`验收标准` 决定研发任务能否独立验收。

没有这个骨架，AI 会按最显眼的东西拆。可拆分 PRD 要把研发任务拆解拉回系统行为和变化边界：状态如何变化，引用如何失效，旧规则如何清理，跨模块如何保持一致，什么算完成。

## 六、从研发任务拆分反推 PRD 模板

如果要改进需求 PRD 模板，我建议在原有“背景、范围、页面、交互、验收”之外，增加这些章节。

### 6.1 类型与变更

`需求类型`：先标明这是新建模块、既有模块增强、概念纠偏、流程替换、数据迁移，还是多模块关系模型变更。不同类型用不同写法。

`变更说明`：如果不是纯新建模块，说明旧能力、旧字段、旧状态、旧接口、旧数据和旧用户路径如何处理，哪些保留，哪些废弃，哪些不兼容。

### 6.2 影响与边界

`影响域`：说明涉及哪些模块、接口、数据模型、运行时链路、旧数据和外部依赖，方便研发任务拆分工具和研发识别是否需要拆成多个研发任务包。

`工程边界`：说明本期不做什么，尤其是不做哪些看起来相关但会扩大实现面的能力。

### 6.3 状态与联动

`状态与规则决议`：集中描述状态机、状态转换、状态对可见性和关联关系的影响。

`跨模块影响`：说明本功能对其他模块的读写关系、自动联动、删除影响和异常状态。

### 6.4 数据与冲突

`数据归属`：说明数据由谁维护、谁只读引用、哪些字段系统生成、哪些字段用户可编辑。

`冲突清理`：列出已废弃的旧描述、确认后变更的规则，以及最终优先生效的口径。

### 6.5 验收与兼容

`行为验收`：把关键验收改成场景化表达。不一定全部写 Given / When / Then，但至少要看出前提、动作和结果。

`发布与兼容`：说明上线前后的兼容变化、迁移影响、异常降级和回滚原则。

### 6.6 原型与拆分

`UI 变更边界`：当 UI 原型由 AI 生成或修改时，记录改动页面、改动组件、保留区域、交互变化、状态展示变化和验收点，避免原型在多轮 AI 修改中漂移。

如果团队已经开始让 AI 根据 PRD 自动拆分研发任务，还应该增加一个拆分检查项：当前 PRD 是否在正确的工程边界内，每条规则是否能落到一个或多个可独立验收的研发任务，跨模块规则是否有专门研发任务承接，异常路径是否有测试覆盖。

这些章节不会让需求人员变成研发人员。它们只是把过去藏在会议、追问和研发经验里的决策，提前写进文档。

## 七、结束语

从研发侧看，AI Coding 没有减少需求和研发之间的协作，只是把协作边界推到了更靠前的位置。

过去的摩擦来自需求不清晰。现在执行速度上去了，不清晰会更快变成错误实现、错误测试和返工。

这次复盘给我的判断是：PRD 规范要调整决策位置，而不只是调整文风。规格化不是重写需求，它要做的是把 PRD 里的隐性边界变成可拆分、可执行、可验收的输入。

几个结论可以先沉淀下来：

- **变化边界易漏**：页面新增通常好理解，旧能力删除、旧规则废弃、关系模型替换、旧数据迁移更容易漏。AI 不会天然知道哪些历史描述已经失效。
- **拆分需要骨架**：目标、非目标、状态规则、跨模块契约、数据归属、行为验收这些信息如果没有结构化，AI 会按最显眼的对象拆研发任务。
- **执行前写边界**：AI 原型和 AI Coding 都需要边界描述。原型初版可以很快，后续局部修改不一定稳定；代码生成也一样。

需求 PRD 和 UI 原型继续承载用户目标、体验路径和页面表达。与此同时，规格化阶段反复补的内容应该尽量前移到 PRD 里：影响域、非目标、状态规则、跨模块影响、数据归属、冲突清理、行为验收和发布兼容。

目标是让需求 PRD 本身达到可拆分研发任务的标准。必要时，一个需求 PRD 可以拆成多个研发任务包，再分别进入研发任务拆解。

这样协作关系会更清楚：需求定义边界，研发把边界规格化并校验拆分质量，AI 执行研发任务，测试验证行为。摩擦能减少，是因为规格化时最容易反复确认的东西，已经提前写进需求 PRD。
