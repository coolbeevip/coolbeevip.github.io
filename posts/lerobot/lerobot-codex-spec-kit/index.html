使用 Codex + Spec-Kit 做出这个 LeRobot Agent PoC

“现在问题解决了，但请你重新 Review 一下今天的几轮修改，看看是不是补丁叠补丁式的修改，如果是的话，重构成最优解”，这是我在这个项目里对 AI 说得最多的一句话。

你也可以说 “review the code, think from greenfield” 或者 “/samplify”，但不管怎么说，这句话的核心意思是：当 AI 在一个已经快速迭代过的代码库里继续修改时，它很容易陷入“补丁叠补丁”的陷阱，而不是停下来重新思考最优解。

更准确地说，这个项目并不是我一个人独立完成的。我主要搭起了大约 70% 的骨架和核心功能，包括命令行入口、配置体系、API 分层、Web 服务、调试面板（dashboard），以及大部分基础测试。到了后期，团队里的同事也逐步参与进来，补上了模型推理、训练以及一些更贴近业务场景的能力。

过去一周时间，我参与做了一个基于 LeRobot 的 SO-101 机械臂 PoC。它最初只是一个“能跑起来”的遥操作脚本，后来逐步长成了一个具备配置驱动、基础测试、服务接口和调试界面的 MVP。

现在这个项目大致长这样：

  • 命令行入口：teleoperate、record、replay、serve
  • Python API 分层：src/api
  • FastAPI 服务：src/webapi
  • React 调试面板：dashboard/
  • 单元测试：tests/unit
  • 配置文件：config/agent.toml

如果只用一句话概括这个项目是怎么做起来的，那就是：

我用 GitHub Spec-Kit 管“要做什么”，用 Codex 管“怎么更快做出来”。

但这不是一个“AI 加上 Spec-Kit 就一路顺风”的故事。中间有很多反复、修正和取舍。Spec-Kit 确实帮了我不少，也确实带来过混乱；Codex 让我省了很多重复劳动，也把我在前端、协作和问题表达上的短板照得很清楚。

这篇文章想讲的，就是这些真实经验。

一、为什么我一开始没有直接写代码

这个项目表面上看像是一个典型的“硬件 + Python”工具，但真正做起来后，我很快意识到它不是一个单点脚本问题，而是一整条链路的问题：

  1. 读取 SO-101 的 follower、leader、camera 配置。
  2. 跑通 teleoperate（遥操作）。
  3. 支持 record（数据采集）和 replay（回放）。
  4. 把能力暴露成 HTTP / WebSocket 服务，方便外部系统调用。
  5. 做一个调试面板，把串口、相机、控制状态、任务状态都可视化。
  6. 后续继续接入推理和 LLM，而不是推翻重来。

如果直接写代码，很容易在第 3 步以后失控。因为你会开始把这些东西揉在一起：

  • 配置解析
  • 硬件连接
  • 任务生命周期
  • Web 接口
  • 前端调试
  • 测试替身和 mock

这些边界一旦不清，后面每加一个功能，都会牵一发而动全身。

所以我一开始没有急着写功能（实际情况是拿到机械臂哪一刻我不知道如何开始），而是先围绕关键功能走 Spec-Kit 的标准流程，把问题拆成几个可以连续交付的阶段。现在回头看，这个决定是对的，因为它很早就把项目骨架定下来了。

二、我怎么用 Spec-Kit 定义这个项目

这并不是我第一次使用 Spec-Kit，我对 Spec-Kit 的理解，不是把它当成一套文档模板，而是把它当成一个“收敛模糊需求”的约束器。

最开始我不是先写实现方案，而是先写清楚系统必须具备什么能力：

  • 用统一配置描述 follower、leader、cameras
  • 支持 teleoperate、record、replay 三个核心动作
  • 代码要有 lint/format 校验，单元测试，git commit hook，保证基本的工程质量
  • 可以在不碰硬件的前提下做单元测试
  • 需要有 Web 层，方便外部系统和前端接入

然后再写约束：

  • 底层依赖 LeRobot
  • Python 版本固定在 3.12
  • 用 uv 管理环境和依赖
  • 配置用 TOML
  • 测试用 pytest
  • 尽量走真实的 LeRobot 路径，而不是再造一层假的 runtime

在具体执行上，我会把不同功能依次推进到 specify -> plan -> tasks 这条链路里。也就是先形成规格，再形成计划，最后拆成任务。最终形成的阶段大致是：

  • 第一阶段：命令行入口跑通 teleoperate、record、replay
  • 第二阶段：把配置、校验、控制会话抽成稳定的 API 层
  • 第三阶段：暴露 FastAPI 接口，补齐状态查询和流式能力
  • 第四阶段：做调试面板，验证整条链路是否真的可操作
  • 第五阶段：接入 LLM / 推理，让这个 PoC 具备更明确的 agent 雏形

这套做法最大的价值，不是“文档更完整”，而是每轮功能落地之后，项目都处在一个能跑、能验的状态。

三、我怎么用 Spec-Kit 的标准流程来驱动 Codex

前面那套拆分思路落地时，我并不是手工写一堆文档，再把文档原样扔给 Codex。而是尽量沿着 github/spec-kit 的官方工作流来推进。

按官方设计，Spec-Kit 的核心流程其实很清晰：

  1. 先用 specify init 把项目初始化成 Spec-Kit 结构。
  2. 用 /speckit.specify 把一个功能想法变成结构化的 Spec-Kit 规格。
  3. 用 /speckit.plan 从这份 Spec-Kit 规格生成实现计划，以及相关的 research、data model、contracts、quickstart 等设计产物。
  4. 用 /speckit.tasks 把计划拆成可执行的任务列表。
  5. 最后再进入 /speckit.implement，或者让 Codex 基于这些产物执行实现。

GitHub 在 spec-driven.md 里对这个流程写得很明确：/speckit.specify 负责把功能描述转成结构化规格；/speckit.plan 负责把规格翻译成技术实现计划；/speckit.tasks 再把计划拆成 tasks.md，供后续执行使用。

也就是说，Spec-Kit 真正提供的不是“几篇文档模板”，而是一条从想法到实现的标准流水线。

对我来说，Codex 在这里不是用来“替我想方案”的，而是用来承接这条流水线的结果。它更擅长的是：

  • 读取已经成型的 Spec-Kit 产物和计划
  • 沿着 tasks.md 执行实现
  • 补充测试、文档和一些结构化重构
  • 在边界清楚时快速推进大量重复性工程工作

所以更准确地说，不是“我把 spec 交给 Codex”，而是“我先用 Spec-Kit 把功能约束成一组 Spec-Kit 产物、计划和任务，再让 Codex 在这组约束内执行”。

这种方式比直接对 AI 说“帮我做个遥操作系统”稳定得多。因为在动手之前，问题定义、边界条件、设计假设和任务拆解基本都已经定下来了。

四、这个项目是怎么一步步长出来的

回头看现在的仓库结构，这个 PoC 的演进路径其实非常清楚。

1. 先把最核心的控制链路做出来

最早我做的不是 Web，也不是前端，而是先把机器人操作闭环打通：

  • teleoperate（遥操作）
  • record（数据采集）
  • replay（回放）

关键不只是“功能跑起来了”，而是我尽早把控制相关的公共能力抽进了 src/api：

  • 配置加载
  • 请求模型
  • 预检校验
  • 控制会话
  • 格式化

这一步很重要。因为如果不提前规定“CLI 只是入口，真正逻辑必须沉到 API 层”，Codex 很容易顺手把所有逻辑都塞进命令行入口里。开始时也许会很快，后面做 Web 服务时就会很难受。

2. 再把“能跑”升级成“能验证”

硬件项目最怕的一件事，就是每次改动都要连机械臂、连相机、连串口才能验证。

所以我很早就让 Codex 帮我做两件事：

  • 把模型和状态边界明确下来
  • 为 API 层和 Web 层补单元测试

这其实是个很实用的策略：先把“可替身”的边界抽出来，再让 Codex 去补测试。边界不清，测试只能测到一整块耦合逻辑；边界清楚以后，补测试这件事就顺手多了。

3. 然后把能力服务化

当 CLI 能稳定工作以后，我没有继续往命令行里堆参数，而是进入了服务化阶段。

于是项目有了：

  • src/webapi/app.py
  • src/webapi/routes_control.py
  • src/webapi/routes_teleoperate.py
  • src/webapi/routes_record.py
  • src/webapi/routes_machine.py
  • src/webapi/routes_chat.py

我当时给 Codex 的要求很明确：

  • Web 层不要直接操作底层细节
  • 状态要可查询
  • 长任务要可启动、可停止、可流式观察
  • 相机流和任务流要区分清楚

最后做出来的效果是，这个项目不仅能“执行机械臂动作”，还能通过 FastAPI 暴露配置读取、串口枚举与探测、摄像头预览、控制会话、任务状态流以及 chat 事件流。

这一步让它从“我自己在终端里能跑”，变成了“其他模块和同事也能接进来验证”。

4. 最后补一个真正能用的调试面板

很多 AI 辅助开发项目都会停在 API 层，理由通常是“前端以后再说”。但我的经验是，如果一个硬件项目没有一个实际可操作的前端调试台，它的可维护性会大打折扣。

这里还有一个更现实的背景：这个项目其实有专门的前端团队，所以从传统分工来看，后端团队原则上不应该自己再做一套前端。放在以前，这件事基本也不可能成立，因为后端团队通常既不擅长，也很难独立把前端真正做起来。

但有了 AI 之后，这件事开始有了现实操作空间。

这里的关键不是“后端团队要替代前端团队”，而是从工程实施的角度，后端团队现在可以为自己补一层足够可用的可视化界面，用来解决联调、排障、设备检查、状态观察这些非常具体的问题。

这类界面和正式产品前端不是一回事。它更像工程工具，不是完整产品。比如调试面板里的“设备检测”页面，本质上就是一个很典型的实施界面：

  • 枚举串口
  • 探测摄像头
  • 验证视频流
  • 观察控制状态
  • 快速判断问题到底出在设备、配置还是服务端

以前这些事情往往只能靠命令行、日志和临时脚本解决；现在借助 AI，后端团队也能把它们做成一个可操作的内嵌前端，这对推进项目很有帮助。

所以我继续让 Codex 把调试面板搭了起来。现在仓库里的 dashboard/ 已经是一个独立的 TypeScript + React + Vite 工程。

这一步不是为了“做个好看的界面”，而是为了验证一个很重要的隐性要求：这个 MVP 至少要做到可观察、可诊断、可操作。

当你真的把串口列表、摄像头探测、会话状态、流式消息都放在一个界面上时，很多原本在 CLI 阶段不明显的问题会立即暴露出来，比如：

  • API 地址不该默认暴露成 0.0.0.0
  • 长任务不能阻塞后续操作
  • 停止任务不应该顺手把底层连接直接断掉
  • 再次创建任务时，旧状态必须被明确回收

这些问题后来很多都体现在代码调整和提交历史里。

不过这里也埋下了另一个问题：当后端团队开始借助 AI 做自己并不熟悉的前端界面时，沟通里的偏差也更容易暴露出来。很多时候我真正想解决的是一个交互目标，但因为我对前端不够熟，描述给 AI 的却成了一个具体但未必正确的实现方案。Wikipedia 甚至为此专门有一个词条 [XY problem][xy-problem]。

五、Spec-Kit 帮了我很多，但它也制造过混乱

如果文章只写“Spec-Kit 很好，Codex 很快”，那是不真实的。这个项目里我至少踩过四类非常典型的问题。

1. Spec-Kit 产物很容易和代码演化脱节

项目进入服务化和前后端联调阶段后，变化速度会比文档更新速度快很多。比如一个接口字段、一个会话状态机、一个 stop 行为的语义，可能在两三次联调里就被改掉了。

如果这时还坚持每个变化都先回写到 Spec-Kit 产物里，再让 Codex 执行，结果往往是：

  • Spec-Kit 产物写了一版
  • 代码又改了一版
  • 最后大家以为 Spec-Kit 产物是真相，但真实行为已经在代码里

我后来解决这个问题的方法很直接：

  • 大阶段设计继续用 Spec-Kit
  • 小步修复以代码和测试为准
  • 等一个阶段稳定后，再统一回收文档和约束

也就是说，我不再要求 Spec-Kit 产物对每个微小变动都实时同步，而是接受它在局部时间窗口里“轻微落后”，但要求阶段结束时重新对齐。

2. MVP 完成后，我主动删掉了 Spec-Kit 产物

AI: “我将按计划实现….并保持向前兼容。"，这会导致你的代码里充斥着各种补丁，最后变成“补丁叠补丁”，而不是“重构成最优解”。

这是我后期做过的一个很重要的动作。

当这个项目的 MVP 已经完成后，我直接删除了 Spec-Kit 产物文件。原因不是我觉得这些过程没有价值，而是它们开始变成大模型的干扰信息。

举个很实际的例子，项目早期我曾经让模型按当时的判断做过一版“基于队列”的功能设计；但随着系统演进，后面我决定把这部分改成异步方法。对人来说，这只是一次正常重构；但对大模型来说，如果旧的计划、任务、设计讨论还完整地留在上下文附近，它就很容易继续把那些已经过时的方案当成“应该遵循的历史意图”。

结果就是：

  • 当前代码已经在往异步方案收敛
  • 旧文档还在强调队列方案
  • 模型每次改动都会下意识往旧路径上补

这时旧的过程文档就不再是资产，而是噪音。

所以我后来形成了一个判断：MVP 阶段结束后，要主动清理那些已经失效的 Spec-Kit 过程产物，只保留仍然有效的约束、接口和最终结论。否则看起来像是在补充上下文，实际上是在反复提醒模型去参考一套已经过时的思路。

3. 用 Spec-Kit 做功能微调，成本偏高

Spec-Kit 很适合回答“要不要做一个 record 能力”，但不太适合回答“这个 record 状态流里还差一个字段，要不要补”。

因为功能微调通常有几个特点：

  • 上下文已经很明确
  • 影响范围通常很小
  • 更像修整而不是设计
  • 决策窗口很短

这种任务如果还要走一遍 Spec-Kit 流水线，经常会出现一种很奇怪的失衡：文档产出比代码改动还大。

所以我后来给自己定了一条简单规则：

  • 涉及架构边界、模块新增、长期接口的变化，走 Spec-Kit
  • 涉及联调修复、字段微调、交互打磨的变化，直接指挥 Codex

比如：

  • 某个 API 返回结构要微调
  • 某个长任务 stop 之后连接不应该被断开
  • 某个调试面板页面联调时地址配置有误

这类任务我后来通常不会再先补 Spec-Kit 产物、再走完整流水线，而是直接让 Codex 改实现、补测试、做回归。

4. Spec-Kit 产物会让人产生“已经想清楚了”的错觉

对于程序员来说，阅读 Spec-Kit 产物实际上是一个痛苦的过程，远没有看代码确定性高，这是我觉得最危险的一点。

当 Spec-Kit 产物写得很完整时，人很容易产生一种幻觉：结构已经清楚了，接下来只需要实现。但硬件项目不是纯文档系统，很多真实问题只有到了联机械臂、联相机、联前端时才会暴露出来，比如：

  • stop 的真实语义到底是停止任务，还是断开连接
  • 长任务到底该阻塞当前会话，还是允许再次创建
  • API 返回值是“理论上合理”，还是“前端真能顺手消费”

这些问题在 Spec-Kit 产物里看上去都很容易，到了真实系统里才发现语义并不稳定。

所以我后来学会把 Spec-Kit 产物当成假设集，而不是真理。Spec-Kit 给我的不是最终答案，而是一组当前阶段比较合理的工程假设。先把它做出来，再用联调和测试去验证；如果验证不通过，就改实现，也改约束，而不是死守原来的 Spec-Kit 产物。

六、和 AI 协作时的 XY Problem

除了 Spec-Kit 本身带来的问题，我后来还发现一个非常典型、而且很容易被忽略的坑：当我对某个领域不够熟时，我会把真正想解决的 Y，错误地描述成 X。

就这个项目和我本人来说，这件事在前端上尤其明显。

有些时候，我真正想解决的是：

  • 页面交互更顺
  • 状态不要阻塞
  • 信息层级更清楚
  • 某个模块更容易操作

但我说给 AI 的却可能是：

  • 把这个按钮挪到左边
  • 给这里加一个弹窗
  • 增加一个轮询
  • 再包一层组件

这些 X 不一定错，但它们未必就是我真正要的 Y。一旦我把问题描述错了，AI 就会很认真地沿着错的方向一直做下去。

这其实就是典型的 XY problem。Wikipedia 对它的描述是：用户提问时，问的是自己尝试的解决方案 X，而不是真正的问题 Y。这个词条本身可以见文末参考链接中的 [XY problem][xy-problem]。

放到 AI 协作里，这个问题比传统的人际协作更容易放大。因为人类同事很多时候会追问一句“你真正想达到的效果是什么”；但 AI 如果收到的是一个结构完整、约束清楚、看起来很像正确任务的描述，往往就直接开始执行了。

所以我后来也给自己加了一条规则，特别是在前端和交互问题上：

  • 先描述真正目标
  • 再描述我以为可行的实现方式
  • 明确告诉 Codex：如果实现方式不对，优先纠正我的问题定义

这其实就是在主动对抗 XY problem。对 AI 来说，最有价值的不是我先给出一个貌似具体的方案，而是先把真正目标说清楚。

七、多人都开始用 AI 之后，协作冲突比以前更容易放大

这个项目后期不是我一个人在推进，而是逐步变成了团队协作。这里我体会特别深的一点是：以前主要是“我和同事协作”，现在很多时候其实变成了“我和我的 AI、同事和他的 AI，再加上我们彼此协作”。

表面上看，大家都更快了；但实际冲突反而更容易变大。

原因是以前两个人写代码，冲突通常发生在实现细节层；现在两个人各自带着 AI 写代码，冲突经常在更早的阶段就出现了，因为 AI 会把每个人当前上下文里的默认假设继续往前推。

典型情况是：

  • 你让 AI 按会话模型继续扩展
  • 同事让 AI 按推理任务模型继续扩展
  • 两边都各自“看起来合理”
  • 合到一起时才发现状态流、接口语义、模块职责已经互相侵入了

我们后来慢慢摸索出的一个更有效的协作方式是：先按模块严格确认边界，再各自实现，最后统一重构。

这里“确认边界”不是口头分工，而是要明确到足够可执行：

  • 谁负责 api 的哪一层
  • 谁负责 webapi 的哪些路由和状态
  • 谁负责模型训练和推理链路
  • 哪些对象是稳定接口，不能随意变更
  • 哪些地方允许先局部实现，最后再统一整理

这样做有两个直接的好处：

  • 每个人都可以在自己的边界内放心使用 AI 提速
  • 系统级冲突被推迟到可控的整合和重构阶段

所以我现在越来越觉得，在 AI 时代里，模块边界比以前更重要。以前边界不清，主要伤的是开发效率；现在边界不清，很容易让多个人各自带着 AI，把系统一起推偏。

八、Codex 在这个项目里真正替我做了什么

对我来说，Codex 真正提高效率的地方主要有四个：

1. 它把 Spec-Kit 流程产出的规格和任务快速翻译成第一版工程骨架

从配置模型、请求对象、命令行参数、API 路由到测试样例，Codex 都能基于已经成型的 Spec-Kit 产物、计划和任务，很快给出一版结构正确的初稿。这样我就不用把时间花在重复劳动上，而是能把精力放在真正的设计决策上。

2. 它很适合做结构化重构

比如把逻辑从命令行入口挪到 api 层，把通用状态收敛到会话对象里，把 Web 层改成统一的 routes/schemas/state 组织方式。这种工作如果纯手工做，非常琐碎；但只要 Spec-Kit 流程已经把边界和任务拆清楚，Codex 执行起来就很快。

3. 它很适合补测试和文档

这个项目里的 README、docs/API.md 以及一系列单测，本质上都属于“高价值但重复度高”的工作。这类任务非常适合让 Codex 跟着已有实现补全。

4. 它能持续保持上下文

项目从 teleoperate 扩展到 record / replay，再扩展到 webapi 和调试面板，如果每次都换一个完全没有上下文的助手，沟通成本会很高。Codex 的优势在于，它能沿着既有代码结构继续往下做，而不是每次都从头理解一遍项目。

九、如果让我再从头做一次

回头看，这个 PoC 能比较顺地从一个 CLI 工具，长成现在这个带服务层、前端层、测试和文档的 MVP，靠的不是我一开始就把所有设计都想对了，而是我采用了一种可以持续修正的工作方式：

  • 先用 Spec-Kit 的标准流程把功能收敛成 Spec-Kit 产物、计划和任务
  • 用 Codex 快速生成初版实现
  • 用测试和真实联调不断收紧接口
  • 再把稳定下来的能力继续往下一层扩展
  • 在多人协作阶段先锁模块边界，再各自借助 AI 实现，最后做统一重构

但如果让我重新来一次，我会比当时更明确地做三件事：

  • 更早清理已经过时的计划、任务和设计过程，避免上下文污染
  • 更早区分“架构问题”和“微调问题”，不要让所有任务都走 Spec-Kit 流水线
  • 在前端和交互问题上，先讲清楚真正目标，再讲我以为的方案，降低 XY problem 的概率

十、最后的总结

如果要我用一句话总结这次实践，那就是：

Spec-Kit 让我先把功能定义清楚，Codex 让我更快把它做出来。

但这句话还不够完整。更完整的版本应该是：

  • Spec-Kit 适合搭骨架，不适合覆盖所有微调
  • Codex 很强，但会被错误上下文和错误问题定义带偏
  • 多人都用 AI 之后，模块边界会比以前更重要

对这个项目来说，AI 不是用来替代工程方法的，它只会把原有的方法放大。方法对了，推进会很快；方法错了，跑偏也会很快。

这也是我现在更相信的一种开发范式：

不是让模型代替工程方法，而是让模型在工程方法之内做执行；同时在必要的时候，主动清理上下文、修正问题定义、重新划清协作边界。
